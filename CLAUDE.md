# TawanFarm App — บันทึกการตรวจสอบบัค

## รีดีไซน์: รายงาน PDF ปิดบ่อ (sec-export)

พี่บ่นว่ากราฟ/ข้อมูลในรายงาน PDF ปิดบ่อ (ที่ปริ้นให้ผู้บริหารดูตัดสินใจ) ดูสับสน ไม่เมคเซนส์ ตรวจแล้วเจอ 3 ปัญหา แก้ครบทุกข้อในรอบเดียว (ข้อ 1 กับ 4 เดิมเสนอแยกกัน แต่เป็นโค้ดจุดเดียวกันเลยแก้พร้อมกัน):
1. **ไม่มีข้อมูลการเงินเลยในรายงาน** — ทั้งที่ตอนปิดรอบเก็บต้นทุน (ลูกกุ้ง/อาหาร/เคมี/แรงงาน/ไฟฟ้า) + กำไรสุทธิไว้ครบใน `harvestResult` แต่ PDF ไม่เคยดึงมาแสดง — เพิ่มฟังก์ชัน `closeSummaryHtml()` ที่จัดกลุ่มข้อมูลเป็น 3 หมวด (🦐 ผลผลิต / 💰 การเงิน / ⏱ การซื้อขาย-เวลา) ใช้ร่วมกันทั้ง preview บนจอและ PDF จริง มีรายได้, ต้นทุนแยกรายการ, รวมต้นทุน, กำไรสุทธิ (สีเขียว/แดงตามบวก/ลบ), กำไร%, ต้นทุนต่อกก.
2. **กราฟยัด 3 แกนไว้กราฟเดียวอ่านยาก** (อาหารสะสม+การกิน+ABW คนละสเกล) — แยกเป็น 2 กราฟเดี่ยวๆ อ่านง่าย: กราฟการเติบโต (ABW) และกราฟอาหารสะสม
3. **กราฟเทียบรอบก่อนหน้าที่มีแค่บนจอ preview ไม่เคยอยู่ใน PDF จริง** — ใส่กราฟนี้ (FCR + อัตรารอด เทียบทุกรอบที่ปิดแล้วของบ่อนั้น) เข้าไปใน PDF ด้วยแล้ว (โชว์เมื่อบ่อนั้นมีรอบปิดแล้ว ≥2 รอบ)
4. `getCloseSummary()` เพิ่ม field การเงิน (costFry/costFeed/costChem/costLabor/costElec/costOther/hasCostData/totalCost/revenue/netProfit/marginPct/costPerKg) ที่ไม่เคยมีมาก่อน
- ทดสอบผ่าน local server ด้วยข้อมูลจำลอง 2 รอบเลี้ยงที่ปิดแล้ว (มีต้นทุน/กำไร/กราฟครบ): preview แสดงข้อมูลการเงินถูกต้อง, genPDF() รันจบไม่มี error, มีกราฟครบ 3 กราฟ (โต/อาหาร/เทียบรอบ) ไม่กระทบข้อมูลจริง

**สถานะ:** ตรวจครบทุกหน้าแล้ว (29/29) เจอบัคจริง 2 จุด แก้ไปแล้วในโค้ด **ยังไม่ได้ deploy** — รอ push ผ่าน `push_*.bat`
- 🔴 #4 ค่าแรงรั่วข้ามรอบเลี้ยง กระทบ Break-even + Risk Exposure Scorecard
- 🔴 #6 ปุ่ม "แก้ไข" หน้าจัดการน้ำสร้างรายการซ้ำแทนอัปเดต

## UX audit — กรอก/แสดงข้อมูลซ้ำซ้อน (แจ้งจากพี่ว่าแอปรู้สึกซับซ้อน)

รายการที่ตรวจเจอ:
1. **[แก้แล้ว]** ปิดรอบเลี้ยง (`openCropModal`) เดิมเคลียร์ต้นทุนลูกกุ้ง (`cm-cost-fry`) ว่างทุกครั้งทั้งที่มีอยู่แล้วใน `p.costFry`, และตั้งต้นทุนอาหาร (`cm-cost-feed`) เป็น 0 เสมอทั้งที่ label เขียนว่า "คำนวณจาก feedLog" (มี `pondFeedCost()` อยู่แล้วแต่ไม่เคยถูกเรียกใช้ตรงนี้) — แก้ให้ดึงค่าจริงมา prefill ให้ทั้งคู่ (ลูกกุ้งยังแก้ไขได้, อาหาร readonly ตามดีไซน์เดิม)
2. **[แก้แล้ว]** ป้าย "FCR" ในแอปมี 2 ความหมายปนกัน — FCR ประเมินระหว่างเลี้ยง (จาก biomass ประเมิน) vs FCR สุดท้ายตอนปิดรอบ (จากน้ำหนักจับจริง) แสดงเป็น "FCR" เฉยๆ เหมือนกันหมดหลายหน้า ทำให้ตัวเลขไม่ตรงกันข้ามหน้าดูเหมือนบัค — ตามที่พี่ขอ **เอา FCR ออกจากทุกหน้ายกเว้นหน้ารายละเอียดบ่อ** (ตัดคอลัมน์ FCR ออกจากหน้าอาหาร + หน้าจัดการบ่อ, ตัดแถวที่ซ้ำกับการ์ด KPI ในหน้ารายละเอียดบ่อเองออก) เหลือแสดงจุดเดียวทั้งแอป ส่วน "FCR สุดท้าย" ตอนปิดรอบ/แก้ไขปิดรอบ (`cm-fcr`/`ecm-fcr`) เป็นคนละค่ากัน (จากน้ำหนักจับจริง) ไม่ได้แตะ
3. **[ปิดประเด็น — ไม่แก้]** เมนู "รายวัน" 6 หน้าย่อย — ทำม็อกอัพ 2 แบบให้ดู (รวมหน้าเดียวมี sub-tab / เช็คลิสต์กดลิงก์ตรง) พี่เทียบกับของเดิมแล้วเห็นว่าของเดิมทำงานสะดวกอยู่แล้ว เพราะน้ำ/อาหารมักไม่ได้กรอกพร้อมกันในเวลาเดียว — ตัดสินใจไม่แก้โครงสร้างเมนูส่วนนี้
4. **[ต่ำ ไม่ต้องแก้]** ชื่อโรงเพาะฟักกรอกได้ทั้งตอนเพิ่ม/แก้บ่อ (`mdl-hatchery`) และตอนปิดรอบ (`ecm-hatchery`) — เป็น snapshot คนละช่วงเวลากันโดยเจตนา ไม่ใช่บัค
5. **[แก้แล้ว — ลบทิ้งทั้งหมด]** ไล่ตรวจหน้าที่เหลือทั้งหมดแล้ว ส่วนใหญ่ไม่พบปัญหาซ้ำซ้อน ยกเว้นจุดนี้: หน้า "ราคาวันนี้" มีฟอร์ม "บันทึกการขายจริง" (`D.sales`) ซ้อนทับกับหน้า "จับกุ้ง" (`D.harvestLog`) เกือบทุกฟิลด์ แต่ไม่เชื่อมข้อมูลกันเลย — พี่แจ้งว่า **ไม่ได้ใช้งานหน้า "ราคาวันนี้" เลย เพราะหาราคามา update ไม่ได้** เลยตัดสินใจ **ลบหน้า "ราคาวันนี้" ทิ้งทั้งหมด**
   - หน้า "Break-even" ใช้ `priceForSize()` จากราคาวันนี้เป็นแกนหลักในการคำนวณ "มูลค่ากุ้งในบ่อ" — ถ้าลบราคาทิ้งหน้านี้จะพังไปด้วย พี่ยืนยันให้ **ลบ Break-even ทิ้งด้วย**
   - หน้า "Risk Exposure Scorecard" ใช้ผลจาก Break-even (`pondBreakeven`) คิด 40% ของคะแนนความเสี่ยงเงินลงทุน — พี่ยืนยันให้ **ลบทิ้งด้วยเช่นกัน** (ไม่ได้ใช้งานหน้านี้)
   - **สรุป: ลบ 3 หน้าออกจากแอปทั้งหมด** — ราคาวันนี้ (`sec-price`), Break-even (`sec-breakeven`), Risk Exposure Scorecard (`sec-riskfin`) พร้อมฟังก์ชัน/เมนู/CSS ที่เกี่ยวข้องทั้งหมด (`priceForSize`, `pondBreakeven`, `riskExposureForPond`, `initPrice/Breakeven/RiskFin`, `renderPrice*`, `saveSale/Price`, `PRICE_SIZES`, `.pr-chart` ฯลฯ) — ฟังก์ชันที่ใช้ร่วมกับหน้าอื่น (`pondFeedCost`, `getLaborCostForPond`, `getElecCostForPeriod`, `wqfeedActivePonds`, `calcRisk` ฯลฯ) เก็บไว้ครบ ไม่กระทบ P&L/ปิดรอบ/หน้าอื่น
   - `D.marketPrices`/`D.sales` เหลือเป็น field เปล่าที่ไม่มีใครอ่าน/เขียนแล้วในโค้ด (ข้อมูลเก่าถ้ามีจะยังอยู่ใน database เฉยๆ ไม่ได้ลบทิ้ง)
   - ทดสอบผ่าน local server แล้ว: ทุกหน้าที่เหลือโหลดได้ปกติ ไม่มี error, ปิดรอบเลี้ยงยังทำงานถูกต้อง (auto-fill ต้นทุนอาหารยังใช้ได้)

แอปจัดการฟาร์มกุ้ง (single-file app: `TawanFarm_App.html` / `index.html`, ~8300 บรรทัด, Firebase backend, deploy ผ่าน GitHub Pages)

## รีดีไซน์: สมุดให้อาหารรายวัน (sec-feed)

หน้า "บันทึกอาหาร" เดิมกรอกได้ทีละบ่อ (เลือกบ่อ→กรอก→บันทึก→สลับบ่อ ซ้ำ 8 รอบ/วัน) เปลี่ยนเป็นตารางกรอกทุกบ่อพร้อมกันในหน้าเดียว:
- ช่อง "อาหารวันนี้" auto-prefill จากค่าล่าสุดที่เคยกรอก (ของวันนั้นถ้ามี ไม่งั้นของวันก่อนหน้าล่าสุด)
- มี date picker ด้านบน — เปลี่ยนวันที่ = โหลด/แก้ไขข้อมูลวันนั้นของทุกบ่อ (ใช้แทนทั้งฟอร์ม "แก้ไข" และตารางประวัติเดิม)
- ปุ่ม "บันทึกทั้งหมด" ครั้งเดียว บันทึก/แก้ไข/ลบ (เว้นว่างช่อง = ลบ) ทุกบ่อพร้อมกัน ตัดสต็อกอาหารอัตโนมัติเหมือนเดิม
- ฟังก์ชันเดิมที่ลบไป: `editFeed`, `saveFeed`, `onFdPond`, `clearFeed`, `populateFeedStockItems`, `updateFeedStockCalc`, `updateFeedRate`, `renderFeedMetrics`, `renderFeedChart`, `renderFeedTable`, `deleteFeed` — แทนที่ด้วย `renderFeedGrid`, `feedGridRow`, `fdGridCalcRate`, `saveFeedGridAll`
- กราฟ FCR/แนวโน้มอาหารรายบ่อแบบละเอียด ย้ายไปดูที่หน้า "รายละเอียดบ่อ" (มีอยู่แล้ว) แทน
- ทดสอบผ่าน local server (`python -m http.server`) แล้ว: บันทึก/แก้ไข/ลบ + ตัดสต็อก/reverse สต็อก ทำงานถูกต้องหมด — การเขียนไป Firebase จริงถูกบล็อกด้วย permission_denied เพราะไม่ได้ login (ยืนยันว่าไม่กระทบข้อมูลจริงระหว่างทดสอบ)

## รีดีไซน์: บันทึกคุณภาพน้ำแบบชิปบาร์ (sec-entry)

หน้า "บันทึกคุณภาพน้ำ" มี 14 พารามิเตอร์/บ่อ — **ไม่ทำเป็นตารางรวมทุกบ่อแบบหน้าอาหาร** เพราะกว้างเกินไปและค่าน้ำเป็นค่าที่ต้องวัดจริง (auto-prefill แบบหน้าอาหารจะอันตราย เสี่ยงบันทึกค่าเก่าทับโดยไม่ได้วัดจริง) แทนที่ dropdown เลือกบ่อด้วยแถบชิปทุกบ่อ + auto-advance หลังบันทึก:
- ก่อนแก้ ทำม็อกอัพโต้ตอบได้ให้ดูก่อน ([mockup_บันทึกคุณภาพน้ำ_ชิปบาร์.html](Tawan Farm/Mockup และ Prototype/mockup_บันทึกคุณภาพน้ำ_ชิปบาร์.html)) ผู้ใช้ยืนยันดีไซน์แล้วค่อยทำจริง
- แถบชิปบนสุดโชว์ทุกบ่อ + สถานะกรอกแล้ว/ยังไม่กรอกของวันที่เลือก (เขียว=กรอกแล้ว, เทา=ยังไม่กรอก, ส้ม=กำลังกรอกอยู่) พร้อม progress bar
- ฟอร์มยังเป็นค่าว่างเสมอเวลาสลับบ่อ — **ไม่ auto-prefill ค่าน้ำ** (คงพฤติกรรมเดิมที่ตั้งใจไว้)
- คลิกชิปบ่อที่กรอกแล้ว → โชว์ฟอร์มว่าง + คำเตือนว่ามีข้อมูลอยู่แล้ว (ไม่ prefill อัตโนมัติ) — การแก้ไขจริงยังต้องกด ✏️ ในตารางประวัติเหมือนเดิม (คงพฤติกรรม safety เดิมไว้ตั้งใจ)
- กด "บันทึก" แล้วเด้งไปบ่อถัดไปที่ยังไม่กรอกอัตโนมัติ ครบทุกบ่อขึ้นข้อความ 🎉
- เปลี่ยนวันที่ (date picker) = ดูสถานะ/กรอกของวันนั้นทุกบ่อ — ใช้แทนการเปิดฟอร์มแก้ไขแยก
- เพิ่มฟังก์ชันใหม่: `renderEpChips`, `updateEpHeader`, `selectEPond`, `epNextUnfilled`, `clearWQFields` (แยกจาก `clearWQ` เพราะของเดิม reset วันที่กลับเป็นวันนี้ทุกครั้งที่สลับบ่อ ซึ่งจะพังตอนแก้ข้อมูลย้อนหลังหลายบ่อ)
- ตารางประวัติของแต่ละบ่อยังอยู่เหมือนเดิม (ไม่ได้ตัดออกเหมือนหน้าอาหาร เพราะ 14 ค่าดูเป็นตารางยังมีประโยชน์)
- ทดสอบผ่าน local server แล้ว: สลับบ่อ/บันทึก/auto-advance/เปลี่ยนวันที่/แก้ไขย้อนหลัง/progress bar ทำงานถูกต้องหมด ไม่กระทบข้อมูลจริง (permission_denied เหมือนครั้งก่อน)

## แก้บัคหน้าโรงเพาะฟัก (sec-hatchery) + เครื่องมือรวมชื่อ

**สาเหตุ:** เจอจากพี่รายงานว่าเว็บจริงหน้า "โรงเพาะฟัก" เห็นชื่อซ้ำ (เช่น "cp ชุมพร1" กับ "ซีพี" แยกกันเป็นคนละแถว)

### 🔴 บัคจริงที่เจอเพิ่ม (แก้แล้ว): `c.closed` ไม่เคยถูกตั้งค่าโดย `confirmCloseCrop()`
`hatcheryStats()` (index.html) และ `hsCropStats()` (หน้าสถิติจับกุ้ง) กรองข้อมูลด้วย `c.closed` แต่ตอนปิดรอบเลี้ยงจริงในแอปไม่เคยตั้งค่า field นี้เลย (ตรวจสอบแล้วว่า field นี้ถูกตั้งเฉพาะใน DEFAULT seed data กับฟังก์ชัน patch ข้อมูลเฉพาะกิจ 1 จุด) — แปลว่ารอบเลี้ยงที่ปิดผ่านแอปตามปกติในอนาคตจะไม่ถูกนับในสองหน้านี้เลย (ต่างจากหน้า P&L ที่เช็คจาก `harvestResult`/`closeDate` แทน เลยไม่มีปัญหานี้) แก้โดย:
- ลบเงื่อนไข `!c.closed` ออกจากทั้ง `hatcheryStats()` และ `hsCropStats()` (การมี record อยู่ใน `cropHistory` ก็หมายถึงปิดแล้วอยู่แล้วโดยธรรมชาติ)
- เพิ่ม `closed:true` ใน `confirmCloseCrop()` เผื่อมีจุดอื่นในอนาคตอ้างอิง field นี้

### เครื่องมือใหม่: 🔧 รวมชื่อโรงเพาะฟักที่สะกดต่างกัน (Admin เท่านั้น)
เพิ่มการ์ดใหม่ท้ายหน้า "โรงเพาะฟัก" (แสดงเฉพาะ `APP_ROLE==='admin'`) — สแกนชื่อโรงเพาะฟักทั้งหมดที่ใช้อยู่จริง (จาก `p.hatchery` ปัจจุบันของทุกบ่อ + `cropHistory[].hatchery` ทุกรอบ) แสดงพร้อมจำนวนรายการที่ใช้ชื่อนั้น ให้ติ๊กเลือกอย่างน้อย 2 ชื่อที่หมายถึงโรงเพาะฟักเดียวกัน กรอกชื่อที่ต้องการใช้จริง กด "รวมชื่อที่เลือก" จะแก้ทุกบ่อ/ทุกรอบที่ใช้ชื่อเหล่านั้นให้เป็นชื่อเดียวกันทันที แล้ว refresh กราฟ/ตารางสรุปให้เอง — ฟังก์ชันใหม่: `hatcheryNameUsage`, `renderHatcheryMergeTool`, `toggleHatcheryMergeSelByIdx`, `mergeHatcheryNames`

**หมายเหตุสำคัญ:** ผมไม่มีบัญชี login เข้าแอปจริง (initFirebase() ทำงานเฉพาะหลัง login เท่านั้น — index.html:7722) เลยมองไม่เห็นข้อมูลจริง 373 รอบที่พี่มี ต้อง**ใช้เครื่องมือนี้เองตอน login แล้ว**ถึงจะรวมชื่อในข้อมูลจริงได้ — ทดสอบ logic ผ่าน local server ด้วยข้อมูลจำลองแล้วว่าทำงานถูกต้อง (merge/คำนวณสถิติใหม่/ซ่อนจาก staff) ไม่กระทบข้อมูลจริง

### 4. 🔴 ค่าแรงรั่วข้ามรอบเลี้ยง กระทบ Break-even + Risk Exposure (แก้แล้ว)
**ไฟล์:** `index.html` และ `TawanFarm_App.html` บรรทัด ~3032 (ฟังก์ชัน `pondLaborCost`)

**บัคคืออะไร:** `pondLaborCost(p)` เดิมรวมค่าแรงจาก `D.laborCosts` "ทุกรายการที่เคยบันทึกให้บ่อนี้" โดยไม่กรองช่วงวันที่เลย:
```js
// เดิม (ผิด)
function pondLaborCost(p){
  return Math.round((D.laborCosts||[]).filter(r=>r.pondId===p.id).reduce((s,r)=>s+(r.amount||0),0));
}
```
แต่ตอน "ปิดรอบเลี้ยง" (`confirmCloseCrop()` บรรทัด ~7595) ระบบจะ archive แล้ว **เคลียร์** `p.records/feedLog/samples/waterLog/chemLog/labResults` ให้บ่อพร้อมเริ่มรอบใหม่ — แต่ **ไม่ได้แตะ `D.laborCosts` เลย** เพราะมันเป็น array กลางที่ผูกกับ `pondId` เฉยๆ ไม่ผูกกับรอบเลี้ยง

**ผลกระทบจริง:** บ่อไหนที่เลี้ยงมาแล้ว ≥2 รอบ (เช่น บ่อ A.1 ปิดรอบ 1 ไปแล้วกำลังเลี้ยงรอบ 2) ตัวเลข "ต้นทุนค่าแรง" ในหน้า **Break-even** (`sec-breakeven`) และ **Risk Exposure Scorecard** (`sec-riskfin`) จะบวกค่าแรงของรอบเก่าที่ปิดไปแล้วเข้ามาด้วย ทำให้:
- ต้นทุนสะสมของรอบปัจจุบันสูงเกินจริง
- สถานะ "คุ้มทุนแล้ว/ยังไม่คุ้มทุน" ผิด (เอียงไปทาง "ยังไม่คุ้มทุน" มากกว่าความจริง)
- % ความเสี่ยงเงินลงทุนในหน้า Risk Exposure สูงเกินจริง

ที่จริงระบบมีฟังก์ชัน `getLaborCostForPond(pondId, fromDate, toDate)` (บรรทัด ~4700) ที่กรองตามช่วงวันที่ถูกต้องอยู่แล้ว และหน้า "ค่าแรง" เองก็ไม่ได้ใช้ `pondLaborCost` แบบผิดนี้ — จุดที่พังคือ `pondBreakeven()` เรียก `pondLaborCost(p)` แบบไม่กรองวันที่เท่านั้น

**วิธีแก้:** เปลี่ยนให้ `pondLaborCost` เรียกใช้ `getLaborCostForPond` แบบกรองตาม `p.startDate` ถึงวันนี้ (ตรงกับ pattern ที่ `getElecCostForPeriodByShare(p.startDate,today(),share)` ใช้อยู่แล้ว):
```js
// ใหม่ (แก้แล้ว)
function pondLaborCost(p){
  return Math.round(getLaborCostForPond(p.id,p.startDate,today()));
}
```
แก้ไปแล้วทั้ง `index.html` และ `TawanFarm_App.html` — **ยังไม่ได้ deploy** (ต้อง push ผ่าน `push_*.bat` ตามขั้นตอนปกติของโปรเจกต์)

**8.1 เบิกออกไม่บังคับเลือกบ่อปลายทาง** — เดิม `saveStockTxBatch()` ตอนโหมด "เบิกออก ไปบ่อ" ไม่เช็คว่าเลือกบ่อหรือยัง ถ้าลืมเลือกแล้วบันทึก สต็อกจะถูกตัดจริงแต่ `pondId` ว่างเปล่า ทำให้รายการหายไปจากหน้า "ประวัติ" (กรองด้วย pondId) และไม่ถูกคิดต้นทุนบ่อไหนเลยตอนปิดบ่อ (`getInventoryCostForPond` กรอง pondId ตรง) — **แก้แล้ว**: เพิ่มเช็ค `if(mode==='out'&&!pondId)` แจ้งเตือนและ block การบันทึก พร้อมใส่ `*` ที่ label "บ่อปลายทาง"

**8.2 ประวัติ "รับเข้า" ดูไม่ได้เลยในแอป** — เดิมหน้า "ประวัติ" โชว์แค่ฝั่งเบิกออกรายบ่อ ส่วนรายการรับเข้าส่วนกลาง (`type==='in'`) ไม่มีที่แสดงเลย มีแค่ตัวเลขนับจำนวนครั้งบน KPI card — **แก้แล้ว**: เพิ่ม toggle ในแท็บ "ประวัติ" แยก "📤 เบิกออก รายบ่อ" / "📥 รับเข้าส่วนกลาง" ฟังก์ชันใหม่ `setStockLogMode()`/`renderStockInLog()` แสดงตารางรับเข้าทั้งหมดเรียงตามวันที่ ลบได้เหมือนกัน

**8.3 ลบรายการประวัติผิด → สต็อกไม่ย้อนกลับ** — เดิม `deleteStockLog()` ลบ log อย่างเดียว ไม่แก้ `item.qty` ให้ (มีแค่ข้อความเตือนว่า "ยอดสต็อกจะไม่ถูกย้อนกลับ") ต้องไปแก้จำนวนคงเหลือเองอีกขั้น เสี่ยงลืม — **แก้แล้ว**: `deleteStockLog()` คำนวณย้อนยอดให้อัตโนมัติ (`out` ที่เคยตัดออก → บวกคืน, `in` ที่เคยรับเข้า → หักออก) ทดสอบแล้วว่าตัวเลขกลับมาถูกต้อง (119→99 ตอนลบ in, 99→100 ตอนลบ out)

ทดสอบทั้ง 3 จุดผ่าน browser + mock data แล้ว ไม่มี console error

### 9. สต็อกพัสดุ — ปรับหมวดหมู่ใหม่ ตัด "ลูกกุ้ง" ออก (แก้แล้ว)
**ไฟล์:** `index.html` และ `TawanFarm_App.html` — `STOCK_CATEGORIES`/`STOCK_CAT_LABEL`/`STOCK_CAT_CSS`/`STOCK_CATALOG` + dropdown 3 จุด (`sk-cat`, `sk-filter-cat`, `ch-hist-cat`)

เปลี่ยนจากเดิม 4 หมวด (ลูกกุ้ง/อาหารกุ้ง/ยา-สารเคมี/อุปกรณ์-วัสดุ) เป็น 7 หมวดใหม่ตามที่แจ้ง: **อาหารกุ้ง, ยากิน, ยาลงน้ำ, จุลินทรีย์กิน, จุลินทรีย์ลงน้ำ, ปูน, อุปกรณ์** — ตัด **ลูกกุ้ง** ออกทั้งหมด เพราะบันทึกซ้ำกับข้อมูลตอนลงกุ้งในหน้าอื่นอยู่แล้ว

- ลบตัวเลือก "ลูกกุ้ง" ออกจาก dropdown ทั้ง 3 จุด และลบ 13 รายการลูกกุ้งออกจาก `STOCK_CATALOG` (แคตตาล็อกราคาไว้กด "นำเข้าสินค้า")
- หมวด "ยา/สารเคมี" เดิม (~90 รายการใน `STOCK_CATALOG`) ถูก**แยกอัตโนมัติ**เป็นยากิน/ยาลงน้ำ/จุลินทรีย์กิน/จุลินทรีย์ลงน้ำ/ปูน โดยดูจากชื่อ/หมายเหตุสินค้า (เช่น มีคำว่า "ปูน" → หมวดปูน, มีคำว่า "จุลินทรีย์/ไบโอ/Bac/Probiotic" → หมวดจุลินทรีย์) — **⚠️ เป็นการแยกแบบประมาณการ ไม่ได้ตรวจสอบกับพี่ทีละรายการ แนะนำให้เข้าไปเช็ค/แก้หมวดของแต่ละสินค้าอีกครั้งในหน้า "รายการพัสดุ" (ปุ่ม ✏️) ให้ตรงกับการใช้งานจริง**
- รายการพัสดุที่เคย**บันทึกไว้จริงในฐานข้อมูล** (ไม่ใช่แค่ในแคตตาล็อก) ที่มีหมวด "ลูกกุ้ง"/"ยา-สารเคมี" แบบเดิม **จะไม่หายหรือพัง** — ยังคงแสดงผลได้ปกติ (label ต่อท้ายด้วย "(เดิม)") เพราะเก็บ mapping หมวดเก่าไว้ใน `STOCK_CAT_LABEL`/`STOCK_CAT_CSS` เพื่อ backward-compat แต่จะเลือกหมวดเดิมพวกนี้ไม่ได้แล้วตอนเพิ่ม/แก้ไขรายการใหม่ — พี่สามารถกดแก้ไขรายการเก่าให้เป็นหมวดใหม่เองได้ทีละรายการเมื่อสะดวก
- ทดสอบผ่าน browser: นำเข้าแคตตาล็อก 105 รายการ ครบทั้ง 7 หมวด ไม่มีหมวด seed หลงเหลือ ไม่มี console error

### 10. ปิดรอบ (openCropModal) — ดึงต้นทุนสารเคมีจากสต็อกจริงอัตโนมัติ (เพิ่มใหม่)
**ไฟล์:** `index.html` และ `TawanFarm_App.html` — `getStockCostBreakdownForPond()` (ฟังก์ชันใหม่) + `openCropModal()` + markup `cropModal` (ช่อง "ต้นทุนสารเคมี")

ตอนปิดรอบ (ปุ่ม "ปิดรอบ" ที่ยังเลี้ยงอยู่ ไม่ใช่แก้ไขรอบที่ปิดแล้ว) ช่อง "ต้นทุนสารเคมี" ตอนนี้**ดึงข้อมูลจริงจากสต็อก**ให้อัตโนมัติแทนที่จะพิมพ์เอง — รวมยอดจากรายการที่เบิกออกจริง (`D.inventoryLog`, `type==='out'`) ของบ่อนั้นในช่วงวันที่เลี้ยง (ไม่รวมหมวด "อาหารกุ้ง" เพราะคิดแยกจาก `pondFeedCost` อยู่แล้ว) พร้อมโชว์บรรทัดสรุปแยกตามหมวด (เช่น "ยาลงน้ำ 2,500฿ · ปูน 1,620฿") ให้เห็นที่มาแบบโปร่งใส ไม่ใช่กล่องดำ — ตัวเลขรวมยังแก้ไขเองได้ถ้าจำเป็น เหมือนกับต้นทุนลูกกุ้ง/อาหารที่ดึงอัตโนมัติอยู่ก่อนแล้ว

**ขอบเขตที่ตั้งใจ:** แก้เฉพาะการ "ปิดรอบปัจจุบัน" (`openCropModal`/`confirmCloseCrop`) เท่านั้น — **ไม่แตะ** โมดัล "แก้ไขรอบที่ปิดแล้ว" (`editCropModal`/`ecm-cost-chem`) และ**ไม่แตะ**ข้อมูลประวัติเก่า `chemLog`/`DEFAULT_CATS` เลยตามที่พี่ระบุไว้ (ข้อมูลรอบที่ปิดไปแล้วให้อยู่เหมือนเดิม)

ทดสอบผ่าน browser + mock data: เบิกพัสดุ 2 หมวด (ยาลงน้ำ 5×500=2,500, ปูน 10×162=1,620) ให้บ่อ A.1 แล้วเปิดโมดัลปิดรอบ ยืนยันว่าช่องต้นทุนสารเคมีขึ้น 4,120 ตรงกับยอดจริง และบรรทัดแยกหมวดถูกต้อง, บ่อที่ไม่มีประวัติเบิกจะเว้นว่างและซ่อนบรรทัดสรุปตามปกติ ไม่มี console error

### 11. หน้าอาหาร (feed grid) — ตัดช่อง "ยี่ห้อ/เบอร์" ออก + ตัวเลือกสต็อกโชว์เฉพาะที่ยังมีของ (แก้แล้ว)
**ไฟล์:** `index.html` และ `TawanFarm_App.html` — `feedGridRow()`/`renderFeedGrid()`/`saveFeedGridAll()`

- ลบช่องกรอก "ยี่ห้อ/เบอร์" (`fdg-*-brand`) ออกจากตารางบันทึกอาหารรายวันทุกบ่อ เพราะข้อมูลสินค้าที่เลือกในช่อง "สต็อกที่ตัด" บอกยี่ห้อ/เบอร์ครบอยู่แล้ว ไม่ต้องพิมพ์ซ้ำ — แต่ยังเก็บฟิลด์ `feedBrand` ในข้อมูลไว้เหมือนเดิม (สำคัญกับตาราง "ประวัติอาหาร" ตอนดูรอบที่ปิดแล้ว) โดยตอนบันทึกจะ**ดึงชื่อสินค้าที่เลือกในช่องสต็อกมาใส่ให้อัตโนมัติแทนการพิมพ์เอง**
- ช่อง "สต็อกที่ตัด" (dropdown เลือกอาหารกุ้งเพื่อตัดสต็อก) ตอนนี้**กรองเฉพาะสินค้าที่ยังมีคงเหลือ (qty>0)** สินค้าที่สต็อกหมดแล้วจะไม่โชว์ในตัวเลือกอีก — ยกเว้นสินค้าที่ถูกเลือกไว้แล้วจากบันทึกเดิม (กันกรณีสต็อกหมดหลังบันทึกไปแล้ว จะยังเห็นค่าที่เคยเลือกอยู่ ไม่หายไปเงียบๆ)
- เพิ่มคอลัมน์ "ผู้จำหน่าย" ต่อจาก "สต็อกที่ตัด" — auto-fill จากฟิลด์ "ผู้จำหน่าย/หมายเหตุ" (`item.note`) ของสินค้าที่เลือกในหน้ารายการพัสดุ อัปเดตอัตโนมัติทันทีที่เปลี่ยนตัวเลือกสต็อก (`fdGridUpdateSupplier()`) เป็นข้อมูลอ่านอย่างเดียว ไม่ใช่ช่องกรอก

ทดสอบผ่าน browser + mock data: ลบคอลัมน์ยี่ห้อ/เบอร์ออกจากตารางแล้ว, สินค้าหมดสต็อก (qty=0) ไม่โชว์ใน dropdown, สินค้ามีของ (qty=50) โชว์ปกติ, บันทึกแล้ว `feedBrand` ถูกเซ็ตอัตโนมัติเป็นชื่อสินค้าที่เลือก, กรณีสินค้าที่เคยเลือกไว้สต็อกหมดทีหลัง (qty เป็น 0) ยังคงโชว์เป็นตัวเลือกที่ถูกซีเล็คอยู่ ไม่มี console error

### 12. เพิ่ม role "CEO" — หน้าภาพรวมผู้บริหาร (เพิ่มใหม่)
**ไฟล์:** `index.html` และ `TawanFarm_App.html` — markup `sec-ceo` (ใน `.main`, ก่อน `sec-overview`), ฟังก์ชันชุด `ceo*`/`renderCeo()`/`showCeoOnly()`/`getStockItemsForPond()`, CSS block "CEO DASHBOARD", แก้ `applyRole()`/`renderUsers()`/`initFirebase()`

ผ่านการรีวิว mockup กับพี่หลายรอบก่อนเริ่มเขียนโค้ดจริง (ดูสรุปดีไซน์ท้ายหัวข้อนี้)

**พฤติกรรม role=ceo:**
- ซ่อนเมนูทั้งหมด (side-nav, แถบแท็บบน, bottom-nav) ด้วย `body.role-ceo` — เห็นแค่หน้า `sec-ceo` เต็มจอ ไม่มีเมนูกรอกข้อมูลรายวันให้เห็นเลย ตรงตามที่ขอ ("ไม่จำเป็นต้องมีปุ่มกดเยอะ")
- `applyRole('ceo')` เรียก `showCeoOnly()` บังคับซ่อนทุก `.sec` แล้วโชว์แค่ `sec-ceo` ทันที ไม่ผ่านกลไก `goTab()` ปกติ (กัน CEO เผลอกดไปหน้าอื่นเพราะไม่มีปุ่มเมนูให้กดอยู่แล้ว)
- ถ้า admin เปลี่ยน role ของ user ที่กำลังใช้งานอยู่ออกจาก "ceo" แบบ real-time (ผ่านหน้าจัดการผู้ใช้) จะพากลับไปหน้าภาพรวมปกติอัตโนมัติแทนที่จะค้างหน้า CEO ที่ตอนนี้ไม่มีสิทธิ์ดูแล้ว (`ceoLeaveIfActive()`)

**หน้า CEO Dashboard มีอะไรบ้าง:**
- KPI 5 ใบ เห็นครบในหน้าเดียว ไม่ต้องเลื่อน: ต้นทุนสะสมรวมทั้งฟาร์ม (hero), อาหารรวมวันนี้, Biomass รวม, บ่อที่กำลังเลี้ยง, บ่อควรติดตาม — **ไม่มีรายได้/กำไรประเมิน** เพราะไม่มีข้อมูลราคาตลาดอ้างอิงจริง (หน้า "ราคาวันนี้" ถูกลบไปแล้วก่อนหน้านี้)
- การ์ดบ่อทั้งหมด (จาก `wqfeedActivePonds()`): sparkline การเติบโตจาก ABW ของ `p.samples`, Biomass/ไซส์/อาหารวันนี้/อาหารต่อแสนตัว (normalize ด้วย `p.stock`), ต้นทุนสะสม, ป้ายสถานะ "ปกติ/ควรติดตาม" (`ceoStatus()` เทียบ ABW ล่าสุดกับ `getAbwTarget(age)` — ต่ำกว่ามาตรฐาน 15%+ ถือว่าควรติดตาม)
- แตะการ์ด → bottom sheet (`ceoOpenSheet()`) โชว์: ข้อมูลบ่อ (วันลงกุ้ง, แหล่งลูกกุ้ง, น้ำหนักรวมสะสม, อาหารรวมสะสมจาก `getCumFeed()`, จัดการน้ำล่าสุดจาก `p.waterLog`), ค่าน้ำเช้าล่าสุดจาก `p.records`, การเติบโต (Biomass/ไซส์/FCR จาก `getFCR()` + กราฟ), และ accordion 3 หัวข้อแบบแตะขยาย (ไม่เปลี่ยนหน้า): **ต้นทุนสะสม**, **ค่าน้ำย้อนหลัง 7 วัน**, **อาหารย้อนหลัง 7 วัน**

**ต้นทุนสะสม (จุดสำคัญ) — ดึงจากสต็อกจริง ไม่ใช่กรอกมือ:**
- ฟังก์ชันใหม่ `getStockItemsForPond(pondId,startDate,endDate)` คืนรายการเบิกสต็อกจริง (`D.inventoryLog`, `type==='out'`, pondId ตรง, ช่วงวันที่เลี้ยง) **ไม่รวมหมวดอาหาร** — itemized ทีละรายการ (ชื่อ/หมวด/วันที่/จำนวน×ราคา) พร้อมจุดสีตามหมวด (`CEO_CAT_DOT`)
- เหตุผลที่ไม่รวมอาหารในรายการ itemized: ต้นทุนอาหารคำนวณจาก `pondFeedCost(p)` อยู่แล้ว (ฟังก์ชันเดิม ใช้ราคาเฉลี่ยสำรองถ้าวันนั้นไม่ได้ผูกกับสต็อกจริง) — ถ้า itemize อาหารซ้ำจะทำให้ต้นทุนอาหารถูกนับซ้ำ/ไม่สอดคล้องกับตัวเลขที่ใช้อยู่แล้วในหน้าปิดรอบ (`openCropModal`)
- ต้นทุนสะสมรวม = `p.costFry` (กรอกตอนลงกุ้ง) + `pondFeedCost(p)` (อาหาร) + ผลรวม stock items (ไม่รวมอาหาร) + `getLaborCostForPond()` (ค่าแรง) + `getElecCostForPeriod()` (ค่าไฟ) — ใช้ฟังก์ชันเดิมที่มีอยู่แล้วทั้งหมด ไม่สร้างวิธีคำนวณคู่ขนานใหม่ที่อาจไม่ตรงกับตัวเลขในหน้าปิดรอบ

**สิทธิ์ในหน้า "จัดการผู้ใช้":** เพิ่มตัวเลือก "CEO (ผู้บริหาร)" ใน dropdown เปลี่ยน role และฟอร์มเพิ่มผู้ใช้ใหม่ — ไม่ต้องแก้ Firebase Console เอง แค่เข้าหน้านี้เลือก role ได้เลยเหมือนเดิม (ตอบคำถามพี่ก่อนเริ่มสร้าง — ดูรายละเอียดในบทสนทนา)

ทดสอบผ่าน browser + mock data ครบ: `applyRole('ceo')` ซ่อนเมนูทั้งหมดถูกต้อง, KPI/การ์ดบ่อ render ตรงข้อมูลจริงที่ seed ไว้ (Biomass, ต้นทุนรวม, itemized stock cost ตรงตัวเลข), accordion เปิด/ปิดถูกต้อง, สลับ role ออกจาก ceo ระหว่างใช้งานพากลับหน้าภาพรวมถูกต้อง, dropdown "จัดการผู้ใช้"/"เพิ่มผู้ใช้ใหม่" มีตัวเลือก CEO ครบ ไม่มี console error

### 62. หน้า CEO — สลับลำดับ "สรุปบ่อ" ขึ้นมาก่อน "ค่าน้ำล่าสุด" ในชีทรายละเอียดบ่อ

### 60. หน้า CEO — ขยายฟอนต์ + ปรับ "สรุปบ่อ" ในชีทรายละเอียดบ่อ

พี่ยืนยัน mockup แล้วว่าการ์ดบ่อหน้าแรก (list) เหมือนเดิมถูกแล้ว ไม่ต้องแก้ แต่ให้ปรับที่ **ชีทรายละเอียดบ่อ** (เปิดตอนกดเข้าไปดูบ่อแต่ละบ่อในหน้า CEO) แทน — ปรับ 2 รอบตามฟีดแบ็ก:
- **หัวชีท** (`.ceo-sheet-meta` — บรรทัด "อายุ X วัน · ABW Y ก." ใต้ชื่อบ่อ): ขยายจาก 11.5px → 15px + ตัวหนาขึ้น ตามที่พี่แจ้งว่ามองไม่เห็น
- **บล็อก "สรุปบ่อ"** (เดิมชื่อ "การเติบโต", `.ceo-stat-grid`): เปลี่ยนจาก 3 คอลัมน์ตัวเล็ก (Biomass/ไซส์/FCR) เป็น 4 ช่อง 2 คอลัมน์ตัวใหญ่ขึ้น (label 10px→12.5px, ค่า 14px→19px) — คงฟิลด์เดิม **Biomass / ไซส์ / FCR ประเมิน** ไว้ตามที่พี่ยืนยัน พร้อมเพิ่ม **Yield/ไร่** (= Biomass ปัจจุบัน ÷ ขนาดบ่อ) เป็นฟิลด์ที่ 4 (ลองสลับเป็นอายุ/น้ำหนักกุ้งไปก่อนรอบแรกเพราะข้อมูลนั้นซ้ำกับหัวชีทอยู่แล้ว พี่ขอให้กลับมาใช้ Biomass/FCR เดิมแทน)
- ต้นทุนสะสม (สรุปรวมต่อสินค้า ไม่โชว่ประวัติรายวัน) ตามที่คุยกันไว้ก่อนหน้า พี่ยืนยันว่าถูกต้องแล้ว ไม่ต้องแก้เพิ่ม
- Mockup ที่ใช้ยืนยันแนวทางเริ่มต้น: `Mockup และ Prototype/mockup_CEO_การ์ดบ่อใหญ่ขึ้น_ต้นทุนสรุปรวม.html`
- ทดสอบผ่าน local server ด้วยข้อมูลจำลอง (บ่อ A.1 อายุ 62 วัน, ABW 20.8g, ไซส์ 48 ตัว/กก., Biomass 8,954 กก./ขนาด 4 ไร่ → Yield 2,239 กก./ไร่): ค่าที่โชว่ในชีทตรงกับที่คำนวณด้วยมือ, ฟอนต์ใหญ่ขึ้นจริงตามที่ตั้ง (หัวชีท 15px, สรุปบ่อ 19px/12.5px), ไม่มี console error

### 59. ปรับเกณฑ์เลือก "บ่อพี่น้อง" ในการ์ดเทียบกับรอบก่อน — เพิ่มความหนาแน่น + แหล่งลูกกุ้ง

เดิม `bmFindSibling()` (ข้อ 57) เลือกบ่อพี่น้องแค่จากขนาดไร่เท่ากัน + อายุ (DOC) ใกล้เคียงที่สุด พี่ขอเพิ่มความหนาแน่นปล่อย (ตัว/ตรม.) และแหล่งลูกกุ้งเข้าไปด้วย เพราะกระทบอัตราการโต/รอดโดยตรง — เปลี่ยนจากเลือกตามอายุใกล้สุดอย่างเดียว เป็นระบบให้คะแนนถ่วงน้ำหนัก 3 ปัจจัย (ขนาดไร่ยังเป็นเงื่อนไขบังคับต้องเท่ากันเหมือนเดิม ไม่ได้เอาเข้าคะแนน):
- ผลต่างอายุ (วัน) ÷ 5
- ผลต่างความหนาแน่น (`dens()` ที่มีอยู่แล้ว) ÷ 10
- แหล่งลูกกุ้งไม่ตรงกัน +3 (เทียบเท่าอายุห่างกัน 15 วัน — ให้น้ำหนักมากสุดเพราะกระทบผลลัพธ์การเลี้ยงมากกว่าอายุ/ความหนาแน่น)
- เลือกบ่อคะแนนรวมต่ำสุด (ยิ่งใกล้เคียงยิ่งดี)
- อัปเดต label ในการ์ดให้โชว์เหตุผลที่เลือกด้วย เช่น "บ่อ A.7 (กำลังเลี้ยง — หนาแน่น 78 ตัว/ตรม., แหล่งลูกกุ้งเดียวกัน)" — พี่จะเห็นว่าทำไมระบบเลือกบ่อนี้มาเทียบ ไม่ใช่กล่องดำ
- ทดสอบผ่าน local server ด้วยข้อมูลจำลอง 3 บ่อพี่น้องที่แข่งกัน: ยืนยันว่าบ่อที่แหล่งลูกกุ้งตรงกัน (แม้อายุห่างกว่านิดหน่อย) ถูกเลือกเหนือบ่อที่อายุใกล้กว่าแต่แหล่งลูกกุ้งต่างกัน ตรงตามน้ำหนักที่ตั้งไว้ ไม่มี console error

### 58. 🔴 แก้บั๊ก: หน้า "ความเสี่ยง" โชว์บ่อที่จับไปแล้ว + พลาดค่า ALK/DO=0

พี่แจ้งว่าหน้า "ประเมินความเสี่ยง" มีบ่อที่จับไปแล้วขึ้นโชว์ข้อมูล — ตรวจสอบเจอ 2 บั๊กจริงในหน้านี้:
1. **`renderRisk()` วนลูป `D.ponds` ตรงๆ ไม่กรองบ่อที่ปิดรอบแล้ว/ยังไม่ลงกุ้งออก** — ทุกหน้าอื่นในแอปที่ต้องเลือก "บ่อที่กำลังเลี้ยงอยู่" ใช้ helper กลาง `wqfeedActivePonds()` (กรอง `pondStatusOv()!=='prep'` ออก) อยู่แล้ว แต่หน้าความเสี่ยงลืมใช้ตัวนี้ ทำให้บ่อที่ปิดรอบไปแล้ว (`startDate`/`records` ถูกล้างเป็นค่าว่าง → `pondStatusOv` คืน `'prep'`) ยังถูกประเมินความเสี่ยงและโชว์การ์ดอยู่ — แก้โดยเปลี่ยนมาใช้ `wqfeedActivePonds()` เหมือนหน้าอื่น พร้อมเพิ่ม empty state ("ยังไม่มีบ่อที่กำลังเลี้ยงอยู่ให้ประเมินความเสี่ยง") กรณีไม่มีบ่อที่กำลังเลี้ยงเหลือเลย — บั๊กนี้กระทบ "บ่อเสี่ยงสูงสุด" ในหน้าภาพรวมด้วย (`renderOvRiskList` อ่านจาก cache เดียวกัน) แก้จุดเดียวได้ผลทั้ง 2 หน้า
2. **`calcRisk()` เช็ค ALK/DO ต่ำด้วย truthy shortcut (`last.alk&&...`, `last.doM&&...`) แทนเช็ค null/ค่าว่างแบบชัดเจน** — ถ้าค่าที่กรอกเป็น `0` พอดี (ALK=0 หรือ DO=0 ซึ่งเป็นกรณีอันตรายที่สุด) เงื่อนไข `0&&...` จะเป็น false เสมอ ทำให้ระบบไม่เตือนความเสี่ยงเลยทั้งที่ควรเตือนหนักสุด — แก้เป็น `last.alk!=null&&last.alk!==''&&last.alk<120` (แพทเทิร์นเดียวกับที่ nh4/no2 ใช้อยู่แล้วในฟังก์ชันเดียวกัน ซึ่งไม่มีบั๊กนี้)
- ทดสอบผ่าน local server ด้วยข้อมูลจำลอง: บ่อที่ปิดรอบแล้ว (stock=0, startDate ว่าง) ไม่ขึ้นในการ์ดความเสี่ยงอีกต่อไป, บ่อที่มี ALK=0/DO=0 ขึ้นเตือน "ALK ต่ำ (0)"/"DO เช้าต่ำ (0)" ถูกต้อง, กรณีไม่มีบ่อที่กำลังเลี้ยงเลยขึ้น empty state ไม่พังหน้า ไม่มี console error

### 57. แจ้งเตือนอากาศ/ข้างขึ้นข้างแรม/วันพระ-วันโกน + การ์ดเทียบกับรอบก่อน (mockup ยืนยันก่อนทำจริงตามที่พี่ขอ)

ต่อจากข้อ 51 (Farm Card) พี่เสนอเพิ่ม 2 เรื่อง แล้วขอให้ทำ mockup ก่อน จากนั้นเพิ่มโจทย์อีก 2 ข้อกลางทาง (วันพระ/วันโกนสำหรับลงแร่ธาตุรวม + แหล่งลูกกุ้ง/อาหารกก.แสนตัวในการ์ดเทียบรอบก่อน) ก่อนอนุมัติให้ implement:

1. **ข้างขึ้นข้างแรม + วันพระ/วันโกน ผสานเข้า Farm Card** (หน้ารายละเอียดบ่อ):
   - `moonPhase()`/`moonInfo()` คำนวณจากสูตร synodic month มาตรฐาน (ไม่ต้องเรียก API เพิ่ม) — โชว์ว่าใกล้จันทร์เต็ม/ดับไหม พร้อมคำอธิบายว่ากินอาหารลดลงช่วงนี้เป็นปกติ
   - `WANPHRA_TABLE` เก็บวันพระจริงของปี 2569 (ตรวจสอบจาก thaipbs.or.th/now/content/3498 ก่อนใช้ ไม่ได้เดาสูตรเอง) — `wanphraInfo()` คำนวณ "ก่อนวันโกน" (2 วันก่อนวันพระ) และ "วันโกน" (1 วันก่อนวันพระ) จากตารางนี้ เพราะพี่แจ้งว่าทุกก่อนวันโกน+วันโกนฟาร์มจะลงแร่ธาตุรวมในบ่อ — มีแถบเลื่อน 10 วันข้างหน้า + แบนเนอร์เตือนถ้าวันนี้/พรุ่งนี้ตรงกับวันที่ต้องลงแร่ธาตุ
   - ⚠️ **ข้อจำกัดสำคัญที่ต้องบอกพี่**: ตารางวันพระใช้ได้แค่ปี 2569 (2026) เท่านั้น เพราะปฏิทินจันทรคติไทยมีปีอธิกมาส/อธิกวารทำให้เดือนไม่สม่ำเสมอ คำนวณเองด้วยสูตรง่ายๆ จะคลาดเคลื่อนได้ — ปีต่อไปต้องมาอัปเดตตาราง `WANPHRA_TABLE` เพิ่ม (หาข้อมูลได้จาก Thai PBS/myhora.com ทุกต้นปี)
   - ผสานสัญญาณเข้ากับ Farm Card เดิม 2 จุด: `pondRiskDO()` เพิ่มเงื่อนไขเตือนถ้าพยากรณ์อากาศพรุ่งนี้ (จาก cache `_wxDaily` ที่ `fetchWx()` โหลดไว้แล้วตอนเปิดแอป ไม่ยิง fetch ซ้ำ) มีฝนหนัก+ลมกระโชก — เพิ่มเป็นเหตุผลเตือน DO ตกล่วงหน้า; `pondRiskFeed()` เติมข้อความอธิบายในเหตุผลเดิมถ้าใกล้ช่วงจันทร์เต็ม/ดับ ว่าอาหารลดอาจเป็นเรื่องปกติ ไม่ใช่สัญญาณโรคเสมอไป (ไม่เปลี่ยนระดับ badge เอง แค่เติมบริบทให้คนอ่าน)

2. **การ์ด "เทียบกับรอบก่อน"** (หน้ารายละเอียดบ่อ ใต้การ์ดเปรียบเทียบ Biomass): ตอบคำถาม "รอบนี้โตช้ากว่ารอบที่แล้วไหม" โดยเทียบที่ **DOC เดียวกัน** ไม่ใช่เทียบวันที่ปัจจุบัน — มี 2 โหมดสลับได้ (ปุ่มไหนมีข้อมูลจริงถึงจะโชว์):
   - **รอบก่อนของบ่อนี้** — ดึงจาก `p.cropHistory` รอบล่าสุดที่ปิดแล้วและมี `samples` (ไม่งั้นไม่มีจุดเทียบ)
   - **บ่อพี่น้องขนาดเดียวกัน** — หาบ่ออื่นที่กำลังเลี้ยงอยู่ (`startDate` มีค่า) ขนาด (`p.size`) เท่ากัน เลือกตัวที่อายุใกล้เคียงกับบ่อปัจจุบันที่สุด (`bmFindSibling()`)

### 55. เพิ่ม "ยอดแถม" แยกจากยอดสั่งซื้อ + การ์ดเปรียบเทียบ Biomass 2 วิธี (แบบฟาร์ม vs แบบเอกสาร)

พี่แจ้งว่าตัวเลขที่ระบบใช้คำนวณอยู่คือยอดสั่งซื้อลูกกุ้งเท่านั้น ไม่รวมยอดแถมจากโรงเพาะฟัก ทำให้ยอดปล่อยจริงในระบบต่ำกว่าความจริง — เพิ่มฟีเจอร์ 2 ส่วน (ผ่านการทำ mockup ยืนยันดีไซน์ก่อนตามที่พี่ขอ):

2. เพิ่ม helper กลาง `pRelease(p)=(p.stock||0)+(p.stockBonus||0)` แล้วแทนที่ทุกจุดคำนวณที่เคยใช้ `p.stock` ตรงๆ ด้วยฟังก์ชันนี้ — ครอบคลุม `dens()`, `getLatestBiomass()`, `fcPeriodFCR()` (FCR ช่วง), feed per 100k ทุกจุด (การ์ดรายละเอียดบ่อ/กราฟสุ่มตัวอย่าง/กราฟเปรียบเทียบ), % กุ้งตายสะสม, เครื่องคิดเลข biomass ในหน้าสุ่มตัวอย่าง, อัตรารอดตอนปิดรอบ (`calcCropEcon`), feed per 100k ในการ์ด CEO dashboard — จุดที่ยังคงอ้างอิง `p.stock` เดิมไว้ตรงๆ คือจุดที่เป็น field กรอกจริง (ฟอร์ม/schema) ไม่ใช่จุดคำนวณ

### 52. รวม CSS badge สถานะ + เชื่อมเกณฑ์หน้าผลแล็บเข้ากับ classifyOv() (2 ข้อที่เลื่อนไว้จากข้อ 50 — แก้แล้ว)
พี่ให้ทำ 2 ข้อที่เลื่อนไว้ต่อได้เลย (ไม่ต้องรอ mockup เพราะเป็นงาน refactor เบื้องหลัง ไม่กระทบหน้าตา):
- **[แก้แล้ว] รวม CSS badge สถานะ 5 ชุดที่เขียนสีซ้ำกัน** (`.b-*`, `.pcb-*`, `.ovd-st-badge.*`, `.ai.*`, `.fc-act-src.*`) — เดิมแต่ละชุดเขียน `background:var(--ob);color:var(--ok)` (และคู่ warn/danger) แยกกันเอง 5 จุด เสี่ยงแก้ที่เดียวไม่ครบ รวมเป็น selector เดียวกัน (เช่น `.b-ok,.pcb-ok,.ovd-st-badge.ok,.ai.ok{...}`) และใช้ `border-left:3px solid currentColor` แทนการเขียนตัวแปรสีซ้ำรอบสามใน `.ai.*` — พบว่า `.b-warn` เดิมใช้ `--w-tx` ส่วนอีก 4 ชุดใช้ `--w` ตรงๆ (ค่าเท่ากันพอดี #B45309 ทั้งคู่ แค่คนละชื่อตัวแปร) รวมเป็น `var(--w)` เดียว ไม่กระทบสี — **ไม่แตะ** `.fc-item-badge.*` เพราะตั้งใจให้เป็นพื้นทึบ+ตัวอักษรขาว (ดีไซน์ต่างจากกลุ่มอื่นโดยตั้งใจ ตามที่ยืนยันไว้ตอนทำ Farm Card)
- **[แก้แล้ว] หน้าผลแล็บ (`updateLabStatus()`) เดิมคำนวณเกณฑ์ warn/danger แยกเองในฟังก์ชัน ไม่ผ่าน `classifyOv()`** — เพิ่ม case ที่ขาดใน `classifyOv()`: `hepatoGreen`/`hepatoYellow` (ใช้เกณฑ์เดียวกับ `tcbsGreen`/`tcbsYellow`), `tvcHepato` (เกณฑ์เดียวกับ `tvc`), `gutInflam`, `lipidDroplet` แล้วแก้ `updateLabStatus()` ให้เรียก `classifyOv()` แทนคำนวณเอง — ระหว่างแก้พบจุดที่เกณฑ์เดิมเพี้ยนกันเล็กน้อย: TVC เดิมในหน้าผลแล็บใช้ `tvc>1e5` (มากกว่าอย่างเดียว) แต่ `classifyOv` ใช้ `tvc>=1e5` (มากกว่าหรือเท่ากับ) — ตอนนี้ตรงกันหมดแล้วผ่าน `classifyOv` เดียว (ผลกระทบจริงมีแค่ค่า TVC ที่เท่ากับ 100,000 พอดีเป๊ะ ซึ่งแทบไม่เกิดในทางปฏิบัติ)
- ทดสอบผ่าน local server แล้ว: badge ทุกกลุ่ม (`.pcb-*` ในการ์ดบ่อ, `.ovd-st-badge` ในตารางสถานะบ่อทั้งหมด, `.fc-item-badge` ใน Farm Card) แสดงสีถูกต้องตรงกับก่อนแก้ 100%, หน้าผลแล็บกรอกค่าทดสอบ (TCBS/ตับ/TVC ตับ/ไขมัน) ได้ผลลัพธ์ตรงตามเกณฑ์เดิมทุกจุด ไม่มี console error

### 51. Farm Card ในหน้ารายละเอียดบ่อ — ความเสี่ยง 5 หมวด + สิ่งที่ต้องทำวันนี้ (สูตรจากเอกสารวิชาการ ดร.พุทธ)
พี่ให้เอกสาร PDF 5 ไฟล์ของ ดร.พุทธ ส่องแสงจินดา (ไนไตรท์สะสม, Pond dynamics, Precision feeding, Shrimp biomass estimation, Shrimp health management — รวม 167 หน้า) ให้อ่านทำความเข้าใจแล้วนำมาปรับใช้ — อ่านครบทุกหน้าด้วยการ render PDF เป็นภาพ (poppler ไม่มีในเครื่อง เลยติดตั้ง pymupdf/Pillow แทน ประกอบภาพเป็น grid 6 หน้า/ภาพเพื่อลดจำนวนรอบอ่าน) สรุปแล้วพี่ให้กลับไปทำต่องานที่ค้างไว้ (Farm Card mockup ข้อ 1 Today's Action + ข้อ 3 ความเสี่ยง 5 หมวด) โดยเอาสูตร/เกณฑ์จากเอกสารมาแทน placeholder logic เดิม:
- **หลักการสำคัญที่สุดจากเอกสาร (ย้ำซ้ำในทุกไฟล์): "แนวโน้มสำคัญกว่าค่าที่วัดครั้งเดียว"** — เพิ่มฟังก์ชัน trend ใหม่ที่ใช้ประวัติที่มีอยู่แล้วในแอป ไม่ต้องมีเซนเซอร์เพิ่ม: `fcDoTrend()` (DO ลดต่อเนื่อง), `fcFeedTrend()` (อาหารกินลดต่อเนื่อง — เอกสารระบุว่านี่คือ "สัญญาณเตือนโรคที่ดีที่สุดอย่างหนึ่ง"), `fcRisingTrend()` (NH3/NO2 มีแนวโน้มขึ้น), `fcPeriodFCR()` (FCR เฉพาะช่วงล่าสุด เทียบกับ FCR สะสม เผยประสิทธิภาพอาหารที่ค่าสะสมอาจกลบไว้)
- **DO Risk**: ใช้ `classifyOv('doM'/'doA')` เดิม + เพิ่มสัญญาณ trend ลดต่อเนื่องเป็น warn แม้ค่าล่าสุดยังไม่ผิดเกณฑ์
- **Pond Stability**: `phSwing` เดิม + เพิ่ม **DO swing** ใหม่ (|doM-doA|, เลียนแบบ phSwing — เกณฑ์ประมาณจากเอกสารที่ระบุว่า pH เสถียร (สวิง≤0.4) สำคัญกว่า pH สูง) + alkalinity/salinity
- **Disease Risk** (จุดใหม่ที่สำคัญสุด): รวมผลแล็บเดิม (TCBS/TVC/EHP/ตับอ่อน) + **สัญญาณเตือนล่วงหน้าจากเอกสาร** — ถ้าอาหารกินลดลง+DO ลดลงพร้อมกัน = danger ทันที (สูตรนี้ย้ำซ้ำ 3 รอบในเอกสารต่างไฟล์กันว่าเป็นชุดสัญญาณเตือนโรคที่แม่นที่สุด) หรือ NH3/NO2 มีแนวโน้มขึ้น = warn
- **Feed Risk**: อาหารกินลดลงต่อเนื่อง + FCR ช่วงล่าสุดสูงกว่า FCR สะสมชัดเจน (>30%)
- **Economic Risk**: FCR สะสม >1.6 warn / >2.0 danger (อิงตัวอย่างตัวเลขจริงจากเอกสาร Precision feeding ที่เทียบ FCR 1.6 vs 1.2)
- แต่ละหมวดคลิกกรองรายการ "สิ่งที่ต้องทำวันนี้" ได้ (เหมือน mockup) ข้อความคำแนะนำอ้างอิงจากเอกสารโดยตรง (เช่น "เปิดเครื่องตีน้ำเพิ่มทันที + ลดอาหาร 20–50%" สำหรับ DO วิกฤต, "ตรวจยอกินอาหารและของเสียในยอ" สำหรับ Feed Risk)
- ติ๊กเสร็จได้ (เก็บสถานะในหน่วยความจำระหว่าง session ไม่บันทึกลง Firebase — เป็นรายการแนะนำรายวัน ไม่ใช่ประวัติถาวร)
- **ตั้งใจไม่แตะ**: เกณฑ์ตี 3–5 (DO ต่ำสุดตามธรรมชาติ) และ Bottom DO (พื้นบ่อต่ำกว่าผิวน้ำมาก) — ทำในซอฟต์แวร์ไม่ได้เพราะไม่มีเซนเซอร์ real-time ตามที่คุยกันไว้ก่อนหน้า, ไม่แก้ threshold เดิมใน `classifyOv()`/`pondStatusOv()` (คงไว้ตามเดิมกันกระทบวงกว้าง เกณฑ์ DO เดิม <3=danger/<4=warn ตรงกับเอกสารอยู่แล้ว), ไม่ทำ "จำนวนกุ้งตายรายวัน" ที่เอกสารแนะนำ (เป็น field ข้อมูลใหม่ กระทบ schema กว้างกว่า รอพี่ยืนยันแยกอีกที)
- อยู่ในหน้ารายละเอียดบ่อ (`sec-ponddetail`) ใต้กล่องแจ้งเตือนเดิม แสดงเฉพาะบ่อที่กำลังเลี้ยงอยู่ (ไม่โชว์ตอนเตรียมบ่อ)
- ทดสอบผ่าน local server ด้วยข้อมูลจำลอง: จำลอง DO+อาหารลดลงต่อเนื่อง 4 วันพร้อมกัน → Disease Risk ขึ้น danger ด้วยข้อความที่ถูกต้อง, จำลอง NO2 ขึ้นอย่างเดียว → Disease Risk ขึ้น warn ถูกหมวด, กดกรองตามหมวด/ติ๊กเช็คลิสต์/แถบความคืบหน้าทำงานถูกต้อง, จอมือถือแสดงผล 2 คอลัมน์อ่านง่าย ไม่มี console error

### 50. รวมสี/ไอคอนสถานะ ok/warn/danger ให้สม่ำเสมอทั้งแอป (audit + แก้แล้ว)
ต่อจากข้อ 49 — พี่ขอทำอีกข้อที่เคยเสนอไว้: สีสถานะให้สม่ำเสมอทั้งแอป ตรวจสอบก่อนพบว่าแอปมีระบบ token สี (`--ok`/`--w`/`--d` + `--ob`/`--wb`/`--db`) อยู่แล้ว แต่ใช้ไม่สม่ำเสมอ 2 แบบ พี่เลือกให้แก้เฉพาะ 2 ข้อนี้ก่อน (เลื่อนข้อ CSS-badge-ซ้ำซ้อน 4 ชุด กับข้อ lab-page คำนวณเกณฑ์แยกจาก `classifyOv()` ไปทำทีหลัง เพราะเสี่ยงกระทบกว้างกว่า):
- **[แก้แล้ว] อีโมจิสถานะไม่ตรงกันข้ามหน้า** — รวมเป็นชุดเดียวทั้งแอป: ✅ (ปกติ) / ⚠️ (เฝ้าระวัง) / 🔴 (อันตราย) แก้ที่: หัวข้อแจ้งเตือนบนสุด (`hdr-alert`), การ์ดรายละเอียดบ่อ, รายการแจ้งเตือนหน้าภาพรวม, การ์ด "สถานะบ่อทั้งหมด" ที่เพิ่งทำในข้อ 49, แบนเนอร์ความเสี่ยงระดับฟาร์ม + สรุปรวมฟาร์ม + การ์ดรายบ่อในหน้าความเสี่ยง (เดิมใช้ 🔴/🟡/⚪ ปนกับ 🔴/⚠️ คนละจุด), สรุปสถานะรวมในหน้าผลแล็บ — **ไม่แตะ** 🟢/🟡 ที่ใช้เป็น "ชื่อสีเชื้อ TCBS" (เช่น "🟢 เชื้อเขียว", "🟡 เชื้อเหลือง") เพราะเป็นศัพท์ทางแล็บจริง (สีโคโลนีบนอาหารเลี้ยงเชื้อ) ไม่ใช่ไอคอนความรุนแรง คนละความหมายกับ ok/warn/danger แม้จะหน้าตาคล้ายกัน — ตรวจแยกทุกจุดก่อนแก้เพื่อไม่ให้ปนกัน
- **[แก้แล้ว] สีเขียนตรงๆ (`#DC2626` ฯลฯ) แทนที่จะใช้ `var(--d)`** — แก้ประมาณ 15 จุด: `.stock-warn`/`.stock-qty-low`/`.wq-danger`/`.wq-arrow-*-ok/-bad` (สต็อก/คุณภาพน้ำ), badge "ใกล้หมด" ในหน้าสต็อก, สี P&L กำไร/ขาดทุน 2 จุด, แถบเตือนบ่อยังไม่กรอกอาหารในหน้าอาหาร (border เดิม `#f0d9a8` เปลี่ยนเป็น `var(--w)` ให้ตรงกับรูปแบบกล่องเตือนที่อื่นในแอป), แถบสีคะแนนความเสี่ยงรายวันในหน้าความเสี่ยง (`#e53e3e`/`#dd6b20`/`#2f855a` → `var(--d)`/`var(--w)`/`var(--ok)`) — ทุกจุดเป็นสีเดียวกันเป๊ะ (hex ตรงกับตัวแปรอยู่แล้ว) ยกเว้นจุดเดียวที่เปลี่ยนโทนจริง (แถบเตือนอาหาร + แถบคะแนนความเสี่ยง) ซึ่งเปลี่ยนให้ตรงกับ pattern ที่ใช้ทั่วแอปอยู่แล้ว ไม่ใช่คิดสีใหม่
- **[ตรวจแล้ว ไม่แก้ — มีเหตุผล]** สี "ต้องดู" บนการ์ด KPI ภาพรวม (`#ff8a70`) จงใจใช้สีส้มอ่อนแทน `var(--d)` เพราะการ์ดนั้นพื้นหลังเข้ม (`--sn-bg2` สีเขียวเข้มธีมไซด์บาร์) `var(--d)` ปกติอาจ contrast ไม่พอบนพื้นเข้ม — เป็นการปรับสีเพื่อ contrast ที่ตั้งใจ ไม่ใช่บัค ไม่แตะ
- **[ตรวจแล้ว ไม่แก้]** สี PARAMS ในหน้าคุณภาพน้ำ (`color:'#e53e3e'` ของ Temp เช้า ฯลฯ) เป็นสีพารามิเตอร์แต่ละตัวสำหรับกราฟ (pH=น้ำเงิน, Temp=แดง, DO=ฟ้า ฯลฯ) คนละแนวคิดกับสีสถานะ ok/warn/danger ไม่ใช่จุดที่ต้องรวม
- ทดสอบผ่าน local server แล้ว: หน้าภาพรวม/ความเสี่ยง/ผลแล็บ แสดงไอคอน/สีใหม่ถูกต้อง, ข้อความ "🟢 เชื้อเขียว" ในผลแล็บยังอยู่ครบไม่ได้ถูกแก้ผิด ไม่มี console error

### 49. หน้าภาพรวม — เพิ่มการ์ด "สถานะบ่อทั้งหมด" รวมสถานะ/เช็คลิสต์/ค่าผิดปกติในตารางเดียว (implement จาก mockup ที่พี่คอนเฟิม)
ต่อจากข้อ 48 — พี่ถามต่อว่ามีอะไรแนะนำอีกให้แอปน่าใช้ขึ้น เสนอ 2 ทาง (สีสถานะสม่ำเสมอทั้งแอป / แดชบอร์ดสรุปภาพรวมสำหรับ admin-CEO) พี่เลือกทำแดชบอร์ดก่อน ทำม็อกอัพให้ดู ([mockup_dashboard_สถานะบ่อทั้งหมด.html](../Mockup และ Prototype/mockup_dashboard_สถานะบ่อทั้งหมด.html)) พี่คอนเฟิมให้เพิ่มเป็นการ์ดใหม่ในหน้าภาพรวม (ไม่ตัดของเดิมออก):
- การ์ดใหม่ "📊 สถานะบ่อทั้งหมด" อยู่ใต้แถบ KPI บนสุดของหน้าภาพรวม — **รวม**ข้อมูลที่ก่อนหน้านี้กระจายอยู่หลายจุด (การ์ดบ่อสีสถานะในกริด, เช็คลิสต์ ✓/– รายวัน, รายการแจ้งเตือนด้านล่าง, บ่อเสี่ยงสูงสุดข้างๆ) มาไว้ในตารางเดียว เรียงบ่อที่ต้องดูก่อน (`danger`→`warn`→`ok`) ไว้บนสุดอัตโนมัติ — ใช้ `pondStatusOv()`/`reasonOv()` เดิมที่มีอยู่แล้ว ไม่สร้าง logic คำนวณสถานะซ้ำ
- แถบสรุป 4 ตัวเลขบนสุด (ต้องดูด่วน/เฝ้าระวัง/ยังไม่กรอกวันนี้/ปกติครบถ้วน) + ชิปกรองด่วน (ทั้งหมด/ต้องดูก่อน/ยังไม่กรอกวันนี้) กดกรองเหลือเฉพาะบ่อที่มีปัญหาได้
- แต่ละแถวโชว์ ✓/– ว่ากรอกน้ำ/อาหาร/สต็อกของวันนี้แล้วหรือยัง (logic เดียวกับ `renderChecklist()`) + เหตุผล/ค่าผิดปกติแบบสั้นในคอลัมน์เดียว คลิกแถวไปหน้ารายละเอียดบ่อได้เลย (`openPondDetail()`)
- Responsive: จอกว้างเป็นตาราง (`.ovd-table`), มือถือ (`max-width:680px`) เปลี่ยนเป็นการ์ดไล่ลง (`.ovd-cards`)
- ฟังก์ชันใหม่: `renderOvDashboard(active)` (เรียกจาก `renderOv()` ท้ายสุด ใช้ `active` ที่คำนวณไว้แล้วไม่คำนวณซ้ำ), `_ovdRows`, `_ovdFiltered`, `_ovdSort`, `setOvdFilter`, `_ovdChk`
- 🔴 เจอบัคระหว่างทดสอบ: ตอนแรกตารางไม่ซ่อนบนมือถือทั้งที่มี media query `.ovd-table{display:none}` แล้ว — สาเหตุคือ selector ของแอปเดิม `.card table{display:block}` (class+element = specificity สูงกว่า `.ovd-table` เฉยๆ) ทับอยู่ แก้โดยเพิ่ม specificity เป็น `.card .ovd-table{display:none}` ในมือถือแทน
- ทดสอบผ่าน local server ด้วยข้อมูลจำลอง 8 บ่อ: เรียงลำดับตามสถานะถูกต้อง, ชิปกรองทำงานถูกต้อง (นับจำนวนตรง), คลิกแถวไปหน้ารายละเอียดบ่อสำเร็จ, จอกว้างแสดงตาราง/มือถือแสดงการ์ดถูกต้องหลังแก้บัค specificity ไม่มี console error

### 48. Micro-feedback ตอนบันทึกสำเร็จ — เพิ่มความน่าดึงดูดในการใช้งาน (implement จาก mockup ที่พี่คอนเฟิม)
พี่ถามว่าทำยังไงให้แอปน่าดึงดูดขึ้นสำหรับ staff/admin — เสนอว่าฟีดแบ็กเชิงฟังก์ชันตอนบันทึกสำเร็จสำคัญกว่าลูกเล่นภาพหวือหวา ทำม็อกอัพ 4 แบบให้ดูก่อน ([mockup_micro_feedback_บันทึกสำเร็จ.html](../Mockup และ Prototype/mockup_micro_feedback_บันทึกสำเร็จ.html)) พี่คอนเฟิมแล้วค่อย implement เข้าแอปจริงทั้ง 4 แบบ:
- **ปุ่ม → เครื่องหมายถูก** (`flashSaveBtn()`) — ปุ่มเปลี่ยนเป็นสีเขียว + ✓ "บันทึกแล้ว" ชั่วครู่แล้วกลับปกติ ใช้กับฟอร์มบันทึกเดี่ยวๆ: ผลแล็บ (`saveLab`), สุ่มตัวอย่าง (`saveSample`), ค่าแรง (`saveLaborCost`), ค่าไฟฟ้า (`saveElecBill`)
- **กล่องข้อความ — เลื่อนเข้า + ไอคอนเด้ง + จางหายนุ่มนวล** — อัปเกรด `showMsg()`, `showElecMsg()`, `showStockMsg()` ทั้ง 3 ฟังก์ชันให้มีทรานซิชันแทนโผล่/หายแบบตัดวูบ (คลาส `.msg-box`/`.msg` ร่วมกัน) มีผลอัตโนมัติกับทุกหน้าที่ใช้ 3 ฟังก์ชันนี้อยู่แล้ว (ผลแล็บ, สุ่มตัวอย่าง, ค่าแรง, ค่าไฟ, สต็อก, คุณภาพน้ำ, จัดการบ่อ, ฯลฯ) — ระหว่างทางพบว่า `.msg` (ใช้โดยค่าไฟ/สต็อก) ไม่เคยมี CSS จริงมาก่อน (กล่องไม่มีสี ไม่มีกรอบ) เลยเพิ่มสไตล์ให้ครบไปด้วยในคราวเดียว
- **Toast มุมจอ** (`showToast()`) — เหมาะกับหน้าที่กรอกต่อเนื่องเร็วๆ ไม่บังฟอร์มเดิม ใช้กับบันทึกคุณภาพน้ำ (`saveWQ`) และสต็อก (`saveStockTxBatch`) — บนมือถือ (`max-width:600px`) เปลี่ยนเป็นแถบเต็มความกว้างเลื่อนขึ้นจากขอบล่างแทนมุมขวา เผื่อ safe-area
- **แถวตารางกะพริบเขียว** (`flashRow()`) — ใช้กับหน้าอาหาร (`saveFeedGridAll`) ทุกแถว `<tr data-pond>` ที่เพิ่งบันทึกจะไฮไลต์เขียวเรียงลำดับทีละแถว (stagger 80ms)
- ทดสอบผ่าน local server ด้วยข้อมูลจำลองทุกฟังก์ชันที่แก้ (saveLab/saveSample/saveLaborCost/saveElecBill/saveWQ/saveStockTxBatch/saveFeedGridAll) ยืนยันว่าปุ่ม/กล่องข้อความ/toast/แถวทำงานถูกต้องครบ ไม่มี console error

### 47. หน้าผลแล็บ — รีดีไซน์ใหม่เป็น 2 แท็บ น้ำ/ตับ-ตัวกุ้ง (implement จาก mockup ที่พี่คอนเฟิม)
ต่อจากข้อ 45 ที่ตรวจเจอว่าการ์ด "บันทึกผลแล็บ" มีช่องกรอกอัดรวมกันเยอะขึ้นเรื่อยๆ (~11 ช่อง) จนเริ่มแน่น — พี่ขอให้ออกแบบหน้าใหม่ให้ทำงานง่ายขึ้นแต่ข้อมูลยังครบทั้งน้ำและตับ ทำม็อกอัพให้ดูก่อน ([mockup_ผลแล็บ_ใหม่_น้ำ-ตับ.html](../Mockup และ Prototype/mockup_ผลแล็บ_ใหม่_น้ำ-ตับ.html)) พี่คอนเฟิมแล้วค่อย implement:
- แบ่งฟอร์มเป็น 2 แท็บสลับได้ (🌊 ตรวจจากน้ำ / 🦐 ตรวจจากตับ-ตัวกุ้ง) แต่ละแท็บมีป้ายนับจำนวนช่องที่กรอกแล้ว (เช่น "2/3") อัปเดตสดตามที่พิมพ์ — สลับแท็บไม่ล้างค่าที่กรอกไว้ในแท็บอื่น กด "บันทึกทั้งหมด" ครั้งเดียวบันทึกข้อมูลทั้ง 2 แท็บพร้อมกันไม่ว่าจะอยู่แท็บไหนตอนกด
- ช่อง "ลักษณะตับอ่อน" (`lb-hepato`) เปลี่ยนดีฟอลต์จาก `normal` (สมมติว่าปกติถ้าไม่ได้กรอก) เป็นว่างเปล่า/`null` (ไม่ฟันธงว่าปกติถ้ายังไม่มีใครตรวจด้วยสายตาจริง) — ตารางประวัติจึงต้องรองรับกรณีนี้ด้วย เพิ่ม badge สีกลาง (`.b-na`) แสดง "-" แทนที่จะขึ้นเป็นแดง/อันตรายทั้งที่ยังไม่มีข้อมูล
- ฟังก์ชันใหม่: `setLabSrc(src,btn)` สลับแท็บ, `updateLabSrcCount(src)` นับช่องที่กรอกแล้วในแท็บนั้น, `onLabInput(src)` เรียกทั้งสถานะรวมเดิม + นับช่องใหม่ทุกครั้งที่พิมพ์ — แก้ `clearLab/initLab/editLab/saveLab/renderLabTable` ให้รองรับดีไซน์ใหม่และดีฟอลต์ `hepato` แบบใหม่
- ทดสอบผ่าน local server ด้วยข้อมูลจำลอง: สลับแท็บค่าที่กรอกไว้ยังอยู่ครบ, ป้ายนับอัปเดตถูกต้อง, บันทึกได้ทั้ง 2 แท็บพร้อมกันไม่ว่าอยู่แท็บไหน, แก้ไขรายการเก่า (`editLab`) prefill ทั้ง 2 แท็บและป้ายนับถูกต้อง, ล้างฟอร์ม (`clearLab`) รีเซ็ตครบรวมถึงป้ายนับกลับเป็น 0, ตารางประวัติแสดง badge "-" สีกลางถูกต้องเมื่อยังไม่ได้ตรวจตับด้วยสายตา และ badge สีตามจริงเมื่อมีการกรอก ไม่มี console error

### 46. สต็อก — เพิ่ม autocomplete ชื่อลูกค้าภายนอก กันพิมพ์ชื่อเดียวกันไม่ตรงกัน (แก้แล้ว)
ต่อจากข้อ 45 ที่ตรวจเจอว่าช่องชื่อลูกค้า/ร้าน (`sk-tx-cust`) เป็นช่องพิมพ์อิสระล้วนๆ เสี่ยงพิมพ์ชื่อเดียวกันไม่ตรงกัน (เช่น "ร้านฟาร์มสุข" vs "ฟาร์มสุข") ทำให้รายงานแยกลูกค้าผิดกลุ่ม — พี่เห็นด้วยกับข้อเสนอให้ทำ autocomplete แทนการเปิดหน้า "จัดการลูกค้า" แยกใหม่ (ไม่อยากเพิ่มตารางข้อมูล/หน้าจัดการเพิ่มโดยไม่จำเป็น):

- ฟังก์ชันใหม่ `populateCustList()` ดึงชื่อลูกค้าที่ไม่ซ้ำจาก `D.inventoryLog` (เฉพาะ `type==='sale'`) มาเรียงตามตัวอักษรไทยใส่ใน datalist — เรียกตอนเข้าโหมด "ขายลูกค้าภายนอก" (`setStockMode('sale')`) และเรียกซ้ำทันทีหลังบันทึกขายสำเร็จ เพื่อให้ชื่อที่เพิ่งพิมพ์ใหม่ใช้ autocomplete ได้ในรายการถัดไปเลยโดยไม่ต้องรีเฟรชหน้า
- ไม่เพิ่ม schema/ตารางใหม่ใดๆ ใช้ข้อมูลที่มีอยู่แล้วล้วนๆ
- ทดสอบผ่าน local server แล้ว: เข้าโหมดขายลูกค้าภายนอกด้วยข้อมูลจำลอง 2 รายการ ลิสต์ขึ้นชื่อครบ 2 ชื่อเรียงตามตัวอักษรถูกต้อง ไม่มี console error

### 45. UX audit รอบ 3 — ข้อความ "ค่าแรง" หลอกตา + เช็คฟีเจอร์ใหม่ช่วงหลังว่าซ้ำซ้อนไหม (แก้แล้ว + ตรวจแล้วไม่พบ)
พี่ขอให้ตรวจอีกรอบว่ามี card ไหนกรอกข้อมูลซ้ำซ้อนกับ card อื่นบ้าง เจอ 1 จุดจริง (ข้อความหลอกตา) + ตรวจลึกฟีเจอร์ใหม่ 4 อย่างที่เพิ่มช่วงหลัง (ขายลูกค้าภายนอก, แพที่จับ, ผู้บันทึกสต็อก, TVC/แบคทีเรียตับ) ไม่พบจุดซ้ำซ้อนเพิ่ม

**[แก้แล้ว] หน้า "ค่าแรง" — ข้อความเชื่อมอัตโนมัติไม่ตรงกับความจริง:** `index.html`/`TawanFarm_App.html` — เดิมมีข้อความเขียนว่า "🔗 เชื่อมกับ 'ปิดรอบ' — ค่าแรงรายบ่อจะถูกดึงไปรวมอัตโนมัติ" แต่โค้ดจริง (`openCropModal()`) เคลียร์ช่อง `cm-cost-labor` ว่างทุกครั้ง เป็นช่องกรอกเองล้วนๆ (ตามที่พี่ยืนยันไว้ก่อนหน้านี้ในเซสชันนี้ว่าต้องการให้กรอกเองเพราะต้องเทียบกับตัวเลขจริงนอกแอป) — ข้อความเดิมไม่เคยถูกอัปเดตให้ตรงกับการตัดสินใจนั้น ทำให้ผู้ใช้เข้าใจผิดว่าไม่ต้องกรอกซ้ำ แก้เป็นข้อความเตือน (สีเหลือง) บอกตรงๆ ว่าต้องกรอกเองตอนปิดรอบ ทดสอบผ่าน browser แล้ว ไม่มี console error

**[ตรวจแล้ว ไม่พบซ้ำซ้อน] ฟีเจอร์ใหม่ 4 อย่างช่วงหลัง:**
- **แพที่จับ** (`hv-buyer` ตอนพาเชียล → `cm-buyer` ตอนปิดรอบ) — ไม่ใช่การพิมพ์ซ้ำ เพราะ `openCropModal()` ดึงชื่อแพจากพาเชียลล่าสุดของรอบมา prefill ให้อัตโนมัติอยู่แล้ว (`lastPartial?.buyer`) แก้ไขได้ถ้าจับที่แพอื่น
- **ผู้บันทึกสต็อก** (`curUserLabel()`) — ดึงจากบัญชี Firebase Auth ที่ล็อกอินอยู่โดยอัตโนมัติ ไม่มีช่องให้กรอกเลย จึงไม่มีทางซ้ำซ้อนกับที่อื่น
- **TVC/แบคทีเรียในตับ** — เช็คแล้วเป็นค่าวัดคนละตัวกับที่มีอยู่แล้วทั้งหมด (TCBS น้ำ/ตับ = จำนวนเชื้อจากตัวอย่างน้ำ/เนื้อเยื่อตับ, TVC น้ำ/ตับ = ค่าวัดคนละมาตรฐาน, "ลักษณะตับอ่อน" = การประเมินด้วยสายตา) ไม่ทับซ้อนกัน แค่ถูกอัดรวมอยู่ในการ์ด "บันทึกผลแล็บ" การ์ดเดียวจนมีช่องกรอกเยอะขึ้นเรื่อยๆ (ตอนนี้ ~11 ช่อง) — ไม่ใช่บั๊ก แต่เป็นจุดที่การ์ดเริ่มแน่นขึ้น ถ้าอยากแบ่งเป็นหมวดย่อย (เช่น accordion) ในอนาคตแจ้งได้
- **ขายลูกค้าภายนอก** — ไม่มีหน้า "จัดการลูกค้า" แยกต่างหาก ชื่อลูกค้า/ร้าน (`sk-tx-cust`) เป็นช่องพิมพ์อิสระไม่มี autocomplete จากชื่อที่เคยกรอกไว้ก่อนหน้า — ไม่ใช่ข้อมูลซ้ำซ้อนกับ card ไหน (ไม่มีชื่อลูกค้าที่อื่นในระบบให้ซ้ำ) แต่เป็นความเสี่ยงพิมพ์ชื่อเดียวกันไม่ตรงกัน (เช่น "ร้านฟาร์มสุข" vs "ฟาร์มสุข") ทำให้รายงานแยกลูกค้าผิดกลุ่มได้ — เป็นข้อเสนอแนะเสริม ไม่ใช่บั๊ก ยังไม่ได้แก้

- ตัด `+(r.costStock||0)` ออกจากสูตรรวม "ต้นทุนรวม" ทั้ง 3 จุด: KPI card, คอลัมน์ "ต้นทุน(฿)" ในตารางรายบ่อ, ผลรวม/สัดส่วนในกราฟโดนัทต้นทุน
- คอลัมน์ "พัสดุ(฿)" ในตารางรายบ่อยังโชว์ `r.costStock` ไว้เหมือนเดิม (เผื่ออยากดูอ้างอิง) แต่เพิ่มไอคอน ℹ️ พร้อม tooltip อธิบายว่าเป็นข้อมูลชุดเดียวกับเวชภัณฑ์/ปูน/อุปกรณ์ ไม่ได้บวกซ้ำเข้าต้นทุนแล้ว
- กราฟโดนัท: เอาหมวด "พัสดุ(ระบบเดิม)" (ที่จริงคือ `costStock` ไม่ใช่ของเก่าอย่างที่ label บอก) ออกจากรายการทั้งหมด เพราะพอไม่รวมในผลรวมแล้วโชว่เป็น slice ก็ไม่มีความหมาย

**ผลกระทบ:** กระทบเฉพาะหน้า P&L (KPI/ตาราง/กราฟ) เท่านั้น — กำไรสุทธิที่โชว์ในหน้านี้ของทุกรอบที่ปิดไปแล้วจะขยับขึ้นเล็กน้อย (เท่ากับส่วนที่เคยนับซ้ำ) ไม่กระทบข้อมูลที่กรอกไว้ในฟอร์มปิดรอบ, PDF สรุปปิดบ่อ, หรือประวัติรอบเลี้ยง เพราะจุดเหล่านั้นไม่เคยใช้ `costStock` มาบวกอยู่แล้ว

ทดสอบผ่าน browser + mock data (จำลองตามตัวอย่างใน mockup เป๊ะ): เบิกพัสดุรวม 4,900 บาท ให้บ่อ, `costMed+costLime+costEquip` ในฟอร์มปิดรอบ = 4,900 เท่ากัน (คนละแหล่งข้อมูลเดียวกัน) → ต้นทุนรวมในหน้า P&L หลังแก้ = 218,100 ตรงกับผลรวม 8 รายการต้นทุนตรงพอดี (ไม่บวกซ้ำ 4,900 อีก) ทั้ง KPI card, ตารางรายบ่อ, และกราฟโดนัทตรงกันหมด ไม่มี console error

### 42. หน้าผลแล็บ — แยก TVC น้ำ/ตับ เหมือน TCBS (เพิ่มใหม่)
พี่แจ้งว่า TVC (Total Viable Count) มีทั้งผลตรวจจากน้ำและจากตับ เดิมมีช่อง TVC เดียวปนกันอยู่ — ทำตามแพทเทิร์นเดียวกับ TCBS/เชื้อเขียว-เหลือง ที่เพิ่งแยกน้ำ/ตับไปก่อนหน้านี้ (ข้อ 36)

**ไฟล์:** `index.html`/`TawanFarm_App.html` — เปลี่ยนป้ายชื่อ `lb-tvc` เป็น "TVC — น้ำ" เพิ่มช่องใหม่ `lb-tvc-hep` ("TVC — ตับ"), แก้ `saveLab()`/`clearLab()`/`editLab()`/`updateLabStatus()`/`renderLabTable()` + เพิ่มตัวเลือกในหน้าเปรียบเทียบ (`cmp-lab-param`/`labLbls`)

- เก็บค่าตับลงฟิลด์ใหม่ `tvcHepato` ใน `p.labResults[]` แยกจาก `tvc` (น้ำ) เดิม ใช้เกณฑ์สีเดียวกัน (ปกติ &lt;10^5)
- ตารางประวัติผลแล็บเพิ่มคอลัมน์ "TVC (ตับ)" แยกจาก "TVC (น้ำ)"
- หน้าเปรียบเทียบเพิ่มตัวเลือก "TVC - ตับ (CFU/g)" แยกจาก "TVC - น้ำ (CFU/mL)" เดิม

ทดสอบผ่าน browser: กรอก TVC น้ำ 2×10^5 + TVC ตับ 5×10^7 บันทึกแล้วค่าถูกต้องครบทั้งคู่, แบนเนอร์สถานะแยกข้อความ "TVC น้ำ"/"TVC ตับ" ถูกต้อง, ตารางประวัติโชว์ทั้งสองคอลัมน์ถูกต้อง, แก้ไขรายการ prefill ค่ากลับมาถูกต้อง, หน้าเปรียบเทียบเลือก "TVC - ตับ" แล้วกราฟขึ้นค่าตรงกับข้อมูลจริงของทั้ง 2 บ่อ ไม่มี console error

### 41. สต็อก — เก็บชื่อผู้บันทึกทุกรายการ (รับเข้า/เบิกออก/ขาย) เอาไว้ re-check ย้อนหลัง (เพิ่มใหม่)
พี่ขอให้เก็บ log ว่าใครเป็นคนบันทึกรายการสต็อกแต่ละรายการ เผื่อกรอกผิดพลาดจะได้ตามหาคนแก้ได้

### 40. สต็อก — ช่องราคาขายลูกค้าภายนอก เลิกดึงราคาแนะนำอัตโนมัติ ต้องกรอกเองทุกครั้ง (แก้แล้ว)

### 38. หน้าภาพรวม — การ์ดสภาพอากาศ เพิ่มข้อมูลน้ำขึ้นน้ำลงของวันนั้น (เพิ่มใหม่)
พี่ขอให้การ์ด "สภาพอากาศวันนี้" ในหน้าภาพรวม (`ov-weather-card`) โชว์น้ำขึ้นน้ำลงของวันนั้นด้วย — ต่อเนื่องจากข้อ 37 ที่เพิ่งเชื่อมตารางน้ำขึ้นน้ำลงจริงเข้าแอป

**ไฟล์:** `index.html`/`TawanFarm_App.html` — ฟังก์ชันใหม่ `_tTodaySummary()` + แก้ `renderOvWeatherCard()`

- `_tTodaySummary()` คืนเวลา/ระดับน้ำขึ้นสูงสุด+ลงต่ำสุดของวันนี้ — ใช้ `TIDE_TABLE` ตรงๆ ถ้าวันนี้อยู่ในช่วงข้อมูลจริง (มิ.ย.–ธ.ค. 2569) ไม่งั้น fallback ไปหา extrema จากโมเดล Harmonic เฉพาะวันนี้เหมือนกลไกในหน้า "น้ำขึ้นน้ำลง" (ข้อ 37)
- เพิ่มบรรทัด "🌊 ขึ้น HH:MM น. (X.X ม.) · ลง HH:MM น. (X.X ม.)" ต่อท้ายการ์ดสภาพอากาศ — แสดงได้ทันทีแม้สภาพอากาศ (`_wxToday`, ดึงจาก API ภายนอก) ยังโหลดไม่เสร็จ เพราะข้อมูลน้ำขึ้นน้ำลงคำนวณในเครื่องไม่ต้องรอ network

ทดสอบผ่าน browser: การ์ดโชว์ "ขึ้น 01:00 น. (2.4 ม.) · ลง 11:00 น. (0.6 ม.)" ตรงกับ `TIDE_TABLE['2026-07-17']` เป๊ะ, ตอนสภาพอากาศยังโหลดไม่เสร็จ (`_wxToday=null`) บรรทัดน้ำขึ้นน้ำลงยังโชว์ปกติ ไม่มี console error

### 37. 🔴 น้ำขึ้นน้ำลง — ค่าที่โชว์ไม่ตรงกับตารางจริงที่พี่ให้ไว้ (แก้แล้ว)
พี่แจ้งว่าข้อมูลน้ำขึ้นน้ำลงในเว็บไม่ตรงกับข้อมูลที่ป้อนให้ — ตรวจแล้วพบว่าหน้า `sec-tide` เดิมคำนวณทุกอย่างจากโมเดล **Harmonic constituent แบบประมาณการ** (`_tAt`/`_tExtrema`, คอมเมนต์เดิมก็ระบุเองว่า "ประมาณ ±30–60 นาที และ ±0.15ม.") ไม่เคยอ่านข้อมูลจริงที่พี่ให้ไว้เลย — เจอไฟล์ตารางน้ำขึ้นน้ำลงจริง 2 ไฟล์ที่ `Mockup และ Prototype/../ตารางน้ำขึ้นน้ำลง/` (`tides_2026-06.json`, `tides_jul-dec_2026.json` รวม 214 วัน มิ.ย.–ธ.ค. 2569 วันละ 1 ขึ้น+1 ลง) ที่ไม่เคยถูกเชื่อมเข้าแอปเลย

**ไฟล์:** `index.html`/`TawanFarm_App.html` — `TIDE_TABLE` (ข้อมูลจริงฝังเป็น const), ฟังก์ชันใหม่ `_tDateKey`/`_tTimeToMs`/`_tAt2`/`_tExtremaMixed`, แก้ `renderTide()`

- ฝังข้อมูลจริงทั้ง 2 ไฟล์ (214 วัน) ลงเป็น `TIDE_TABLE` ในโค้ดโดยตรง
- `_tExtremaMixed(s0,s1)` แทนที่ `_tExtrema()` เดิมในจุดที่ใช้แสดงรายการขึ้น/ลงรายวัน — วันไหนอยู่ในช่วงตารางจริงใช้ค่า `hiT/hiV/loT/loV` ตรงๆ (ไม่ผ่านการคำนวณเลย ตรงเป๊ะกับที่ให้มา) วันไหนอยู่นอกช่วง (ก่อน 1 มิ.ย. 69 หรือหลัง 31 ธ.ค. 69) ค่อย fallback ไปสแกนหา extrema จากโมเดล Harmonic เหมือนเดิม
- `_tAt2(ms)` แทนที่ `_tAt()` เดิมสำหรับ "ระดับน้ำตอนนี้" และกราฟคลื่นรายวัน — ตารางจริงมีแค่ 1 ขึ้น+1 ลง/วัน (ไม่ใช่เส้นโค้งต่อเนื่อง) จึงต้อง interpolate ระหว่างจุดขึ้น/ลงจริงของวันก่อน/วันนี้/วันถัดไป (สูงสุด 6 จุด) ด้วยสูตร cosine ease ให้เส้นโค้งลื่นและวิ่งผ่านค่าจริงพอดีทุกจุด นอกช่วงตารางจริง fallback ไป `_tAt()` เดิม
- แก้ข้อความแจ้งท้ายการ์ดให้ตรงกับความจริง: ถ้า 7 วันข้างหน้าอยู่ในช่วงตารางจริงครบ โชว์ "✅ ข้อมูลจริงจากตารางน้ำขึ้นน้ำลง" แทนคำเตือน "ค่าโดยประมาณ" เดิม (ถ้าคาบเกี่ยวช่วงนอกตาราง โชว่าข้อความผสมบอกตามจริง)

ทดสอบผ่าน browser: รายการขึ้น/ลง 7 วันข้างหน้าตรงกับ `TIDE_TABLE` เป๊ะทุกวัน (เวลา+ระดับน้ำ), ระดับน้ำ ณ เวลาที่ตรงกับจุดขึ้น/ลงจริงในตารางให้ค่าตรงเป๊ะ (ทดสอบ 17 ก.ค. 69: ขึ้น 01:00=2.40ม., ลง 11:00=0.60ม. ตรงกับตารางทุกตัว), วันที่นอกช่วงตาราง (ทดสอบ มี.ค. 2570) ยัง fallback ไปใช้ค่า Harmonic เดิมถูกต้อง ไม่มี console error

### 35. PDF ปิดบ่อ — เพิ่มติ๊กเลือกกราฟที่ต้องการแสดง (เพิ่มใหม่)
พี่ขอให้มีช่องติ๊กเลือกว่าจะโชว์กราฟตัวไหนบ้างใน PDF สรุปปิดบ่อ จากข้อมูลที่มีบันทึกไว้จริง

**ไฟล์:** `index.html`/`TawanFarm_App.html` — markup `#ex-chart-toggles` ในหน้า `sec-export`, ฟังก์ชันใหม่ `renderExChartToggles()`, แก้ `onExPondChange()`/`renderExPreview()`/`genPDF()`

กราฟที่มีให้เลือกต่างกันตามโหมดที่เลือก (ตามที่ `genPDF()` ออกแบบไว้เดิม):
- **เลือกบ่อเดียว** (สรุปหน้าเดียว): 💰 กำไร-ต้นทุนเทียบรอบก่อนหน้า, 📊 FCR/อัตรารอดเทียบรอบก่อนหน้า — โชว์ก็ต่อเมื่อบ่อนั้นมีรอบปิดแล้ว ≥1 รอบ
- **เลือกทุกบ่อ** (สรุปละเอียดหลายหน้า): 📈 การเติบโต (ABW), 🍚 อาหารสะสมตลอดรอบ, 📊 FCR/อัตรารอดเทียบรอบก่อนหน้า — แต่ละอันโชว์ก็ต่อเมื่อมีอย่างน้อย 1 บ่อที่มีข้อมูลจริงสำหรับกราฟนั้น (สุ่มตัวอย่าง/feedLog/ปิดรอบ ≥2 รอบ ตามลำดับ)

`renderExChartToggles()` ซ่อนแถวติ๊กที่ไม่มีข้อมูลให้ใช้อัตโนมัติ (ไม่โชว์ตัวเลือกที่กดไปแล้วไม่มีผลอะไร) ค่าเริ่มต้นติ๊กไว้ครบทุกอันที่มีข้อมูล สลับบ่อ/โหมดใหม่จะรีเซ็ตกลับเป็นค่าเริ่มต้น (ไม่จำการติ๊กข้ามบ่อ) — `genPDF()` เช็คสถานะติ๊กผ่าน `chartWant(kind)` ก่อนจะ push chart job และวาดกล่อง canvas ในทุกจุดที่เคยสร้างกราฟแบบไม่มีเงื่อนไขมาก่อน ถ้าไม่ติ๊กเลยสักอันในโหมดบ่อเดียว ส่วน "กราฟสำหรับผู้บริหาร" จะหายไปทั้งบล็อก (grid ปรับเป็น 1 คอลัมน์อัตโนมัติถ้าติ๊กแค่อันเดียว)

ทดสอบผ่าน browser + mock data: สลับโหมดบ่อเดียว/ทุกบ่อ โชว์ชุดติ๊กถูกต้องตามที่มีข้อมูลจริง, ปลดติ๊ก "FCR เทียบรอบก่อนหน้า" แล้วสร้าง PDF (สปายที่ `Chart` constructor) ยืนยันว่าสร้างเฉพาะกราฟกำไร-ต้นทุนจริง ไม่มีกราฟ FCR ปนมา, สลับไปโหมดทุกบ่อปลดติ๊ก "การเติบโต" ยืนยันไม่สร้างกราฟ ABW แต่กราฟเปรียบเทียบยังสร้างตามปกติ ไม่มี console error

### 34. 🔴 "แพที่จับ" ในหน้าสรุป/PDF ปิดบ่อ ไม่มีที่มาจริง โชว์ค่างว่างเสมอ (แก้แล้ว)
พี่ถามว่าฟอร์ม "แก้ไขข้อมูลปิดบ่อ" ไม่มีช่องกรอกแพที่จับกุ้งเลย แล้วหน้าสรุป PDF ที่มีแถว "ปิดบ่อกับแพ" ดึงข้อมูลมาจากไหน — ตรวจแล้วพบว่า**ไม่มีที่มาจริงเลย เป็นค่าว่างเสมอ**

**สาเหตุ:** หน้า "จับกุ้ง" มีช่อง "แพที่ทำการจับ" (`hv-buyer`) แต่โผล่**เฉพาะโหมดพาเชียลเท่านั้น** พอเลือก "ปิดรอบ" จะซ่อนฟอร์มนี้ทันทีไปเปิด `cropModal`/`editCropModal` แทน ซึ่ง**ไม่มีช่องแพเลยทั้งคู่** — และ `getCloseSummary()` (index.html:6735) เดิม hardcode `buyer:''` ในแถว "ปิดบ่อ" ตรงๆ เลย ทำให้แถว "ปิดบ่อกับแพ" ในหน้าสรุป/PDF (index.html:6675, 6808) โชว์ "-" เสมอสำหรับทุกรอบที่ปิดผ่านแอปตามปกติ

**วิธีแก้:** เพิ่มช่อง "แพที่จับ / ผู้ซื้อ" ในฟอร์มปิดรอบทั้งสอง (`cm-buyer` ในฟอร์มปิดรอบใหม่, `ecm-buyer` ในฟอร์มแก้ไขรอบที่ปิดแล้ว) เก็บลง `harvestResult.buyer` แล้วแก้ `getCloseSummary()` ให้ `closeRows` อ่านจาก `harvestResult.buyer` แทนการ hardcode ค่าว่าง — ฟอร์มปิดรอบใหม่ (`openCropModal`) ยัง**ดึงชื่อแพจากพาเชียลล่าสุดของรอบนั้นมา prefill ให้อัตโนมัติ** (เผื่อจับที่แพเดิม) แก้ไขได้ถ้าไม่ตรง ตามแนวทางเดียวกับที่ต้นทุนลูกกุ้ง/อาหารดึงอัตโนมัติอยู่แล้ว

ทดสอบผ่าน browser: ปิดรอบผ่านปุ่มจริงในหน้า "รอบเลี้ยง" กรอกแพ "แพสมชายซีฟู้ด" → บันทึกลง `harvestResult.buyer` ถูกต้อง, `getCloseSummary()` อ่านค่าคืนมาถูกต้อง, เปิดฟอร์มแก้ไขรอบที่ปิดแล้ว prefill ชื่อแพเดิมถูกต้อง แก้เป็นชื่อใหม่บันทึกซ้ำได้ถูกต้อง (round-trip ครบ) ไม่มี console error — **ข้อมูลรอบเก่าที่ปิดไปแล้วก่อนแก้ครั้งนี้จะยังโชว์ "-" เหมือนเดิม เพราะไม่เคยมีการเก็บชื่อแพไว้จริงๆ ตั้งแต่ต้น ต้องเข้าไปกรอกย้อนหลังผ่านฟอร์ม "แก้ไขข้อมูลปิดบ่อ" เอาเองทีละรอบถ้าต้องการ**

### 33. บ่อที่ถูกลบ — ประวัติเบิกสต็อกโชว์ id ดิบแทนชื่อ (แก้ต้นตอ) + รีดีไซน์หน้า "ประวัติ เบิกออกพัสดุ" เป็นตารางเดียวกรองได้ (แก้แล้ว)
พี่รายงานว่าตารางประวัติหน้า "สารเคมี" โชว์ชื่อบ่อแปลกๆ แบบ `P1781065709969` — ตรวจแล้วนั่นคือ pondId ภายใน ไม่ใช่ชื่อบ่อ

**ต้นตอ:** `delPond()` (index.html, หน้า "จัดการบ่อ") ลบบ่อออกจาก `D.ponds` โดยไม่เก็บชื่อไว้เลย ส่วนประวัติเบิกสต็อกเก่า (`D.inventoryLog`) ที่เคยผูกกับ `pondId` นั้นยังอยู่ (ไม่ได้ลบตาม เพราะเป็นข้อมูลบัญชี/ต้นทุนที่ต้องเก็บไว้) — ทุกจุดที่ join ชื่อบ่อจาก pondId (เช่น `renderChemHistory()` เดิมใช้ `pondMap[r.pondId]||r.pondId`) เลยหาไม่เจอแล้ว fallback ไปโชว์ id ดิบแทน

**แก้ต้นตอ:** 
- `delPond()` เก็บชื่อบ่อไว้ใน `D.deletedPondNames={id: name}` ก่อนลบออกจาก `D.ponds`
- เพิ่มฟังก์ชันกลาง `pondNameById(pondId)` เดียว — เช็คบ่อที่ยังอยู่ก่อน ถ้าไม่เจอเช็ค `D.deletedPondNames` (ต่อท้าย "(ลบแล้ว)" ให้รู้ทันที) ถ้ายังไม่เจอค่อย fallback เป็น id ดิบ — แก้ `renderChemHistory()` ให้ใช้ฟังก์ชันนี้แทน `pondMap` เดิม

**รีดีไซน์หน้า "ประวัติ" → "เบิกออก รายบ่อ" ในหน้าสต็อก:** พี่แจ้งว่าอยากได้แบบดูง่าย/อัปเดตง่ายกว่าเดิม ไม่ต้องกดเข้าไปทีละบ่อ — ของเดิม (`renderStockPondGrid`/`showStockPondHist`/`backToStockPondGrid`) เป็นกริดการ์ดบ่อ (เห็นเฉพาะบ่อที่ยังเลี้ยงอยู่ด้วย — บ่อปิด/ลบแล้วมองไม่เห็นประวัติเลย) ต้องกดเข้าไปทีละบ่อถึงเห็นรายการ **ลบฟังก์ชันเหล่านี้ทิ้งทั้งหมด** แทนที่ด้วยตารางเดียว (`renderStockOutLog()`) รวมทุกบ่อ (รวมบ่อลบแล้ว) พร้อมตัวกรอง 3 ช่อง: เลือกบ่อ (dropdown สร้างจากบ่อที่มีประวัติจริงเท่านั้น ใช้ `pondNameById` โชว์ชื่อ), หมวดพัสดุ, เดือน — กดปุ่ม 🗑️ ลบได้ในแถวทันทีเหมือนเดิม (ย้อนสต็อกคืนอัตโนมัติ + รีเฟรชตัวกรอง)

ทดสอบผ่าน browser + mock data: บ่อที่ถูกลบ (`delPond()`) โชว์ "ชื่อบ่อเดิม (ลบแล้ว)" ถูกต้องในทั้ง dropdown และตาราง (ไม่ใช่ id ดิบ), กรองตามบ่อ/หมวด/เดือนได้ถูกต้อง, ลบแถวแล้วสต็อกย้อนคืนถูกต้อง + dropdown ตัดตัวเลือกบ่อที่ไม่มีประวัติเหลือออกอัตโนมัติ, สลับโหมด เบิกออก/รับเข้า ยังทำงานปกติ ไม่กระทบแท็บขายลูกค้าภายนอก (ข้อ 32) ไม่มี console error

### 31. % ผู้จัดการ — ตัดสูตร % ออก เหลือช่องกรอกเองล้วนๆ (ยังไม่ผูกสูตร)
พี่แจ้งว่า "%ผู้จัดการ" ต้องดึงข้อมูลจากไฟล์อื่นนอกแอป ตอนนี้ยังไม่ต้องผูกสูตรอะไรเลย — ทำแบบเดียวกับที่ทำให้ "%คนเลี้ยง" ในข้อ 30 (รวมช่อง "% ผู้จัดการ" + "จำนวนเงินผู้จัดการ" (readonly, เดิมคำนวณจาก % × กำไรก่อนหักส่วนแบ่ง) เหลือช่องเดียวชื่อ "ส่วนแบ่งผู้จัดการ (บาท)" กรอกยอดเงินตรงๆ เอง ไม่มีการคำนวณ/placeholder แนะนำใดๆ (ต่างจาก %คนเลี้ยงที่ยังมีตัวเลขแนะนำเป็นเงา — อันนี้ไม่มีสูตรให้แนะนำเลยตามที่พี่บอก)

**ผลต่อสูตรกำไรสุทธิ:** เดิม `กำไรสุทธิ = กำไรก่อนหักส่วนแบ่ง − ส่วนแบ่งผู้จัดการ(auto)` ตอนนี้ทั้งคนเลี้ยงและผู้จัดการเป็นยอดเงินกรอกเองล้วนเหมือนต้นทุนรายการอื่นๆ แล้ว เลยง่ายลงเป็น `กำไรสุทธิ = รายได้ − ต้นทุนตรงทั้งหมด (รวมส่วนแบ่งทั้งสองเป็นรายการต้นทุนปกติ)` — ตัด concept "กำไรก่อนหักส่วนแบ่ง" ออกทั้งฟอร์มปิดรอบและฟอร์มแก้ไข

**โครงสร้างข้อมูล:** ตัดฟิลด์ `pctManager` ออกจากการเก็บข้อมูลใหม่ (รอบเก่าที่เคยมีอยู่ยังอ่านแสดงผลได้) เหลือ `amtManager` เก็บยอดยืนยันตรงๆ เหมือน `amtCaretaker` — พบและแก้บัคแบบเดียวกับข้อ 30 ไปพร้อมกัน: `closeSummaryHtml` เดิมเช็คเงื่อนไขโชว์แถวจาก `pctManager!=null` ทำให้รอบใหม่ (ไม่มี pctManager แล้ว) จะไม่โชว์แถวส่วนแบ่งผู้จัดการเลย — แก้เป็นเช็คจาก `amtManager!=null` แทน (จุดเดียวกับที่เคยแก้ให้คนเลี้ยงไปแล้ว)

ทดสอบผ่าน browser จริง: ฟอร์มปิดรอบเหลือช่องเดียว "ส่วนแบ่งผู้จัดการ (บาท)" ไม่มี placeholder แนะนำ (ตามตั้งใจ) → กรอกยอดตรงๆ คำนวณกำไรสุทธิถูกต้องตรงสูตรมือ 100% (ทดสอบ: รายได้ 520,000 − ต้นทุนรวมทุกรายการรวมคนเลี้ยง 11,000+ผู้จัดการ 7,000 = กำไร 222,000 ตรงเป๊ะ) → ปิดรอบจริงบันทึก `amtManager` ถูกต้อง ไม่มี `pctManager` ติดไปด้วย → `closeSummaryHtml` โชว์แถวส่วนแบ่งผู้จัดการถูกต้องแม้ไม่มี % (7,000 บาท ไม่มีต่อท้าย %) → เปิดฟอร์มแก้ไขรอบที่ปิดแล้ว แก้ยอดจาก 7,000→9,000 กำไรลดลง 2,000 ตรงตามจริง บันทึกถูก — ไม่มี console error, sync สองไฟล์แล้ว **ยังไม่ push**

### 30. ส่วนแบ่งคนเลี้ยง + ค่าจับกุ้ง — เปลี่ยนจากกรอกเองล้วนๆ เป็นคำนวณแนะนำจากไซส์/น้ำหนักจับ
พี่ให้เงื่อนไขคำนวณจริงมา 2 จุด (จากเดิมที่เป็นช่องกรอกเองล้วนๆ ในข้อ 29) — คุยชัดก่อนทำ 3 คำถาม (ตัดช่อง "%คนเลี้ยง" เดิมแล้วเก็บเป็นช่องเดียวชื่อ "%คนเลี้ยง" แต่ค่าในช่องเป็น "บาท" ไม่ใช่ % / ช่วงไซส์ 91-99 ที่ไม่ระบุไว้ให้นับเท่าช่วง 90-70 / ช่วง 24-20 กับ 19 ลงไป อัตราเท่ากันจริง 3 บาท/กก.)

**สูตร:**
- `caretakerRateBySize(size)`: ไซส์ ≥100 ตัว/กก.=ไม่ได้ (0), 70-99=1 บ./กก., 50-69=1.5, 30-49=2, 25-29=2.5, ≤24=3 — คูณกับน้ำหนักที่จับได้ (กก.) เป็นยอดแนะนำ
- `catchCostByWeight(weightKg)`: 1.5 บาท/กก. ปกติ, ถ้าน้ำหนักจับไม่ถึง 4,000 กก. (4 ตัน) คิดเหมา 6,000 บาทแทน

**UX "เงาตัวเลข" ตามที่ขอ:** ช่อง "% คนเลี้ยง (บาท)" กับ "ค่าจับกุ้ง (บาท)" ไม่เติมค่าอัตโนมัติให้ — คำนวณแล้วโชว์เป็น **placeholder** (เช่น "แนะนำ 9,000 (2 บ./กก.)") ในช่องที่ยังว่างเปล่า ต้อง**พิมพ์ยืนยันเอง** (จะปัดเศษเป็นเลขกลมๆ ได้ตามใจ) ค่าที่ใช้คำนวณจริงคือค่าที่พิมพ์เท่านั้น ไม่ใช่ตัวเลขแนะนำ

**โครงสร้างข้อมูล:** ตัดฟิลด์ `pctCaretaker`/ช่อง "จำนวนเงินคนเลี้ยง" แยกออกทิ้ง เหลือฟิลด์เดียว `amtCaretaker` (เก็บยอดเงินที่ยืนยันแล้วตรงๆ) — รอบเก่าที่เคยมี `pctCaretaker`+`amtCaretaker` คู่กันยังอ่านแสดงผลได้ปกติ (`closeSummaryHtml` เช็คเงื่อนไขใหม่จาก `amtCaretaker!=null` แทน `pctCaretaker!=null` ไม่งั้นรอบใหม่ที่ไม่มี pctCaretaker จะไม่โชว์แถวนี้เลย) — "% ผู้จัดการ" ไม่แตะ ยังเป็น % กรอกเอง × กำไรก่อนหักส่วนแบ่งเหมือนเดิม

**🔴 บัคที่เจอระหว่างทดสอบและแก้ทันที (เกิดจากงานนี้เอง ไม่ใช่ของเดิม):** ตอนย้ายยอดเงินคนเลี้ยงจาก "% × กำไร" มาเป็น "ยอดเงินตรง" ลืมเอาค่านี้ไปรวมใน `totalCostDirect` ก่อนคำนวณกำไรก่อนหักส่วนแบ่ง — ทำให้ฐานคำนวณ % ผู้จัดการสูงเกินจริงเท่ากับยอดคนเลี้ยง (เช่น คนเลี้ยง 9,000 × %ผู้จัดการ 3% = ผู้จัดการเพี้ยนไป 270 บาท) เจอจากการเทียบเลขคำนวณมือกับที่แอปคำนวณจริงไม่ตรงกัน แก้แล้วโดยรวม `cm-pct-caretaker`/`ecm-pct-caretaker` เข้า `totalCostDirect` ก่อน (edit-form เดิมทำถูกอยู่แล้ว บัคอยู่ที่ close-form เท่านั้น)

ทดสอบผ่าน browser จริง: unit-test สูตร `caretakerRateBySize`/`catchCostByWeight` ครบทุกช่วงไซส์+น้ำหนัก (17+5 เคส ผ่านหมด รวมขอบเขตที่ยืนยันเป็นพิเศษ 91-99 และ 24-20/19ลงไป) → เปิดฟอร์มปิดรอบ placeholder ขึ้นถูกต้อง (ไซส์ 40→2บ./กก.×4,500กก.=9,000, น้ำหนัก 4,500กก.≥4ตัน→1.5×4,500=6,750) → ช่องยังว่างจริงไม่ใช่ค่าเติม → พิมพ์ยืนยันเลขกลมๆ เองแล้วคำนวณกำไร/ส่วนแบ่งผู้จัดการตรงตามสูตรมือ 100% (แก้บัคข้างต้นแล้วตรง) → ปิดรอบจริง บันทึกถูก → getCloseSummary/closeSummaryHtml/PDF/P&L อ่านค่าตรงกันหมด → เปิดฟอร์มแก้ไขรอบที่ปิดแล้วค่าขึ้นครบถูกต้อง — ไม่มี console error, sync สองไฟล์แล้ว **ยังไม่ push**

### 29. ตัวเลขปิดบ่อ — แยกต้นทุนเป็น 12 หมวดละเอียด + ส่วนแบ่งกำไรคนเลี้ยง/ผู้จัดการ
พี่ต้องการแยกต้นทุนตอนปิดบ่อจากเดิม 6 หมวด (ลูกกุ้ง/อาหาร/สารเคมี/ค่าแรง/ค่าไฟ/อื่นๆ) เป็น 12 หมวดละเอียด — คุยชัดเจน 3 จุดก่อนทำ (ค่าจับกุ้งกรอกเองเหมือนข้ออื่น, %ส่วนแบ่งเป็น%ของกำไรสุทธิและหักเป็นต้นทุน, ค่าแรงเปลี่ยนจากดึงอัตโนมัติเป็นกรอกเองทั้งหมด — ค่าไฟฟ้ายังคงสูตร 2,500 บ./วันเหมือนเดิม)

**หมวดใหม่ (เรียงตามที่พี่ระบุ):**
1. ต้นทุนอาหารกุ้ง — auto (`pondFeedCost`)
2. ต้นทุนลูกกุ้ง — auto จาก `p.costFry` แก้ไขได้
3. ต้นทุนเวชภัณฑ์ — auto ใหม่ รวมยากิน+ยาลงน้ำ+จุลินทรีย์กิน+จุลินทรีย์ลงน้ำ (และหมวด "เดิม" ยา/สารเคมี ถ้ามี) จากสต็อกจริง ผ่านฟังก์ชันใหม่ `getStockCostGroupsForPond()`
4. ต้นทุนปูน — auto ใหม่ จากสต็อกหมวด lime
5. ต้นทุนค่าแรง — **เปลี่ยนเป็นกรอกเองทั้งหมด** (เดิมดึงอัตโนมัติจาก `getLaborCostForPond`) ตามที่พี่ยืนยัน
6. ต้นทุนค่าไฟฟ้า — auto เหมือนเดิม (2,500 บ./วัน)
7. ต้นทุนน้ำมัน — กรอกเอง (ใหม่)
8. อุปกรณ์ — auto ใหม่ จากสต็อกหมวด equip
9. อุปกรณ์นอก — กรอกเอง (ใหม่)
10. ค่าจับกุ้ง — กรอกเอง (ใหม่)
11-12. % คนเลี้ยง / % ผู้จัดการ — กรอกเอง คำนวณเป็นจำนวนเงินอัตโนมัติ (readonly), **สูตร**: `กำไรก่อนหักส่วนแบ่ง = รายได้ − ต้นทุนตรงทั้ง 10 หมวด` แล้ว `จำนวนเงินแต่ละส่วน = กำไรก่อนหักส่วนแบ่ง × %` สุดท้าย `กำไรสุทธิ = กำไรก่อนหักส่วนแบ่ง − ส่วนแบ่งคนเลี้ยง − ส่วนแบ่งผู้จัดการ` (หลีกเลี่ยงการคำนวณวนซ้ำ — ใช้ "กำไรก่อนหักส่วนแบ่ง" เป็นฐานคูณ % ไม่ใช่กำไรสุทธิสุดท้าย)

**ความเข้ากันได้กับข้อมูลเก่า:** ไม่ลบ `costChem`/`costOther` (หมวดเดิม) ออกจากระบบ — `getCloseSummary()` ยังอ่านรวมเข้า `totalCost` ได้ครบ, แสดงเป็นแถว "(เดิม)" ใน `closeSummaryHtml()`/PDF/ตารางแก้ไขเฉพาะตอนที่รอบนั้นมีค่าอยู่จริง (รอบใหม่ที่ใช้หมวดละเอียดจะไม่เห็นแถวนี้เลย) — ฟอร์ม "แก้ไขข้อมูลปิดบ่อ" (`editCropModal`) ยังคงช่อง "สารเคมี(เดิม)/อื่นๆ(เดิม)" ไว้ท้ายฟอร์มเผื่อต้องแก้รอบเก่าที่เคยกรอกแบบเดิม

**จุดที่แก้ครบทุกจุดที่ใช้ต้นทุนปิดบ่อ:** ฟอร์มปิดรอบ (`cropModal`)+`calcCropEcon()`+`confirmCloseCrop()`, ฟอร์มแก้ไข/เพิ่มข้อมูลเก่า (`editCropModal`)+`openEditCropModal()`+`saveEditCrop()`, `getCloseSummary()`, `closeSummaryHtml()` (พรีวิวหน้าจอ), PDF ปิดบ่อหน้าเดียว (`pdfCostBreakdownHtml()` + กราฟ "กำไร-ต้นทุนเทียบรอบก่อนหน้า" ใน `genPDF()`), หน้า P&L ทั้งหมด (`renderPLKPI`/`renderPLTable`/`renderPLCostChart`/`exportPLCSV`)

**🔴 บัคเดิมที่เจอระหว่างทดสอบ (มีอยู่ก่อนแล้ว ไม่ได้เกิดจากงานนี้ — ยังไม่ได้แก้ รอพี่ตัดสินใจ):** หน้า P&L (`renderPLKPI`/`renderPLTable`) บวก `costStock` (จาก `getStockCostForPond`, สรุปยอดเบิกพัสดุทั้งหมดของบ่อ) เพิ่มเข้าไปเป็นบรรทัดแยก "พัสดุ(฿)" **ทั้งที่ข้อมูลเดียวกันนี้ถูกนับไปแล้วใน costMed/costLime/costEquip (เดิมคือ costChem)** — ทำให้ต้นทุนรวมในหน้า P&L สูงเกินจริง (นับซ้ำ) เจอจากการทดสอบ: บ่อทดสอบมีสต็อกเบิกจริง 4,900 บาท ต้นทุนตรงรวม (ไม่รวม costStock) คำนวณถูกต้อง 317,508 แต่ตาราง P&L โชว์ 322,408 (317,508+4,900 ซ้ำ) — ปัญหานี้มีมาตั้งแต่ก่อนแยกหมวด (costChem เดิมก็ดึงจากแหล่งเดียวกับ costStock) เป็นปัญหาเชิงโครงสร้างของหน้า P&L ที่ควรตัด costStock ออก หรือเปลี่ยนไปใช้แทนที่ costMed/costLime/costEquip แทนการบวกซ้อน — **ไม่ได้แก้ในรอบนี้เพราะกระทบตัวเลข P&L ย้อนหลังทุกรอบที่เคยปิดไปแล้ว ต้องคุยกับพี่ก่อน**

**บัคที่เจอระหว่างทำและแก้แล้ว (สำคัญ — เจอจากการทดสอบ ไม่ใช่แค่เขียนโค้ดแล้วจบ):**
1. **CSS comment ไม่ปิด `*/`** — เขียนคอมเมนต์อธิบาย admin-only แบบหลายบรรทัดแล้วลืมปิด ทำให้ browser กลืนกฎ CSS ที่ตามมาทั้งหมดเป็นคอมเมนต์ยาว 1,546 ตัวอักษร รวมกฎ admin-only ใหม่ 3 กฎ **และกฎเดิมที่ไม่เกี่ยวข้องเลยอีกหลายตัว** (`.cat-tag`, `.kpi-card`, `.ov-kpi-bar` ฯลฯ) — เจอจากการ scan `document.styleSheets` จริงในเบราว์เซอร์เทียบจำนวน rule ก่อน/หลังแก้ (542 → 558) ไม่ใช่แค่ syntax-check ด้วย Node (Node เช็ค JS เท่านั้น ไม่เช็ค CSS)

2. **[ร้ายแรง] เช็คลิสต์เตรียมบ่อฟื้นคืนหลังล้าง** — `resetPrepPond()`/ปิดรอบ ตั้ง `prepChecklist=null` แต่ `mergeChecklist(null, remote)` คืนค่า remote → เช็คลิสต์เดิมกลับมาหลัง sync (ปุ่ม 🗑️ ที่เพิ่งทำในข้อ 21-22 ใช้กับ Firebase จริงไม่ได้ผลถาวร!) — แก้โดย: เช็คลิสต์มี `id` ตั้งแต่สร้าง (`prep_<pid>_<ts>`), ของเก่า normalizeD backfill เป็น `prep_<pondId>` (deterministic เท่ากันทุกเครื่อง), ตอนล้างแผน/ปิดรอบ push id เข้า deletedIds, `mergeRemote` เช็ค tombstone ก่อน merge เช็คลิสต์ (ฝั่งไหนตายแล้วห้ามฟื้น), `mergeChecklist` คง id ไว้ในผลลัพธ์เสมอ

### 24. รวมระบบ "จับกุ้ง/ปิดบ่อ/ประวัติ" ให้เหลือชุดข้อมูลเดียว หน้าเดียว (รีดีไซน์ใหญ่ — แก้แล้ว)

### 23. รอบเลี้ยง — เพิ่มปุ่ม "➕ เพิ่มข้อมูล" สำหรับกรอกรอบเก่าที่ตกหล่น (เพิ่มใหม่)

- **โหมดเพิ่ม** (`idx===-1`): validate ต้องเลือกบ่อ+กรอกรอบที่+ปีครบ, เตือนซ้ำถ้าบ่อนั้นมีรอบที่+ปีเดียวกันอยู่แล้ว (ยังยืนยันเพิ่มซ้ำได้ถ้าต้องการ), สร้าง object รอบใหม่ทั้งชุด (records/feedLog/samples ฯลฯ เป็น array ว่างเพราะไม่มีข้อมูลรายวันของรอบเก่า) push เข้า `p.cropHistory`

### 22. ปุ่มล้างแผนเตรียมบ่อ — แก้จุดสัมผัสเล็กเกินไป กดบนมือถือไม่ติด (แก้แล้ว)
พี่แจ้งว่ากดปุ่ม 🗑️ บนการ์ดเตรียมบ่อ (หน้าภาพรวม) แล้ว "ไม่มีอะไรเกิดขึ้นเลย" — ตรวจสอบด้วยการยิง `.click()` จริงบนปุ่ม (ไม่ใช่เรียกฟังก์ชันตรงๆ ซึ่งไม่เจอบั๊กแบบนี้) พบว่าโค้ดทำงานถูกต้อง 100% แต่**จุดสัมผัสจริงเล็กมากแค่ 17×13px** (เพราะตั้ง `padding:0;line-height:1` ตอนสร้างข้อ 21 ไม่ได้คิดเผื่อขนาดนิ้วสัมผัสจริงบนมือถือ) เล็กกว่ามาตรฐานจุดสัมผัสมือถือ (~44px) มาก ทำให้กดพลาดบ่อยหรือกดไม่ติดเลย

**แก้แล้ว:** เพิ่ม `padding:6px;margin:-6px` (ขยายพื้นที่กดโดยไม่ดันเลย์เอาต์รอบข้าง) + `min-width:28px;min-height:28px;display:inline-flex` จัดไอคอนกึ่งกลาง จุดสัมผัสจริงตอนนี้ ~31×28px

ทดสอบยืนยันด้วย `getBoundingClientRect()` ก่อน/หลังแก้ (17×13 → 31×28) และยิง `.click()` จริงอีกครั้งว่ายังล้าง `prepChecklist` ได้ถูกต้อง ไม่มี console error — **บทเรียนสำคัญ:** ต่อไปนี้เวลาเพิ่มปุ่มเล็กๆ (icon-only) ต้องเช็คขนาดจุดสัมผัสจริงด้วย ไม่ใช่แค่ทดสอบว่าฟังก์ชันทำงานถูกหรือเปล่า

### 21. ภาพรวม: ปุ่ม "เปิดบ่อใหม่" + ล้างแผนเตรียมบ่อ, จัดการบ่อ: ตัดตั้งค่ารอบเลี้ยงต่อปี (เพิ่มใหม่/แก้แล้ว)

- ลบการ์ดตั้งค่า "จำนวนรอบเลี้ยงต่อปี" ออกจากหน้า "จัดการบ่อ" ตามที่พี่บอกว่าไม่จำเป็น พร้อมลบฟังก์ชัน `updateCropsPerYear()` และการ populate ค่าใน `populateAll()` ที่กลายเป็น dead code (ค่า `D.settings.cropsPerYear` เดิมยังใช้ในลอจิกขึ้นรอบใหม่ตอนปิดรอบอยู่ แค่ไม่มี UI ให้เปลี่ยนค่าแล้ว ค้างที่ค่าเดิม/default 2)

**บั๊ก 2 จุดที่เจอระหว่างทำ (แก้แล้ว):**
1. ปุ่ม "เปิดบ่อใหม่" ตอนแรกใช้แค่ `class="admin-only"` — แต่ `.home-quick button{display:flex}` (specificity class+type = 0,0,1,1) **สูงกว่า** `.admin-only{display:none}` เฉยๆ (0,0,1,0) ทำให้ปุ่มโชว์ทุก role ไม่ใช่แค่ admin ทั้งที่ยังไม่ทันแก้ role อื่น — แก้โดยเพิ่ม `.home-quick .admin-only{display:none!important}` + คงกฎเดิม `body.role-admin .home-quick .admin-only{display:flex!important}` ไว้คู่กัน (specificity สูงกว่าเลยชนะตอน role=admin)
2. หลังแก้ข้อ 1 พบอีกจุด: `body.role-admin .admin-only{display:revert!important}` (กฎเดิมของทั้งระบบ) รีเซ็ตกลับเป็นค่า default ของ browser (`inline-block` สำหรับ `<button>`) ไม่ใช่ `display:flex` ที่ปุ่มนี้ต้องการ ทำให้ layout ไอคอน+label เพี้ยนตอนโชว์ให้ admin — ต้องระบุทับเฉพาะจุดนี้ด้วย `body.role-admin .home-quick .admin-only{display:flex!important}`

ทดสอบผ่าน browser ครบ: role staff → ปุ่มทั้ง 2 จุด (เปิดบ่อใหม่ + ล้างแผนเตรียมบ่อ) ซ่อนถูกต้อง (`display:none`), role admin → โชว์ถูกต้องด้วย display ที่ถูกต้อง (`flex`/`block` ตามจุด), `resetPrepPond()` ล้าง prepChecklist เป็น null จริง ไม่กระทบ D.ponds, ไม่มี console error

### 20. เพิ่ม dropdown เลือกปี "ประวัติการจับกุ้ง" (เพิ่มใหม่)
เพิ่ม `<select id="hv-filter-year">` ข้าง dropdown เลือกบ่อ — ค่าเริ่มต้นคือปีปัจจุบัน (พ.ศ.) ไม่โชว์ทุกปีรวมกันตามที่พี่ขอ ("เยอะเกินไป") มีตัวเลือก "ทุกปี" ให้เลือกดูรวมได้ถ้าต้องการ ตัวเลือกปีสร้างจากปีที่มีข้อมูลจริงใน `D.harvestLog` (บวกปีปัจจุบันเสมอแม้ยังไม่มีข้อมูล)

**บั๊กที่เจอระหว่างทำ (แก้แล้ว):** รอบแรกวาง logic populate dropdown ไว้ใน `populateAll()` เหมือน `hv-filter-pond` — พอทดสอบพบว่ากดปุ่ม "ซิงค์จากรอบเลี้ยง" (ข้อ 19) แล้ว dropdown ปีไม่อัปเดตทันที เพราะปุ่มนั้นเรียกแค่ `renderHarvest()` ไม่เรียก `populateAll()` — ย้าย logic มาไว้ใน `renderHarvest()` เอง คำนวณรายการปีใหม่ทุกครั้งที่ render แทน

**บั๊กที่สอง (แก้แล้ว):** ตอนคง state ค่าที่เลือกไว้ข้าม render ใช้ `hvYearEl.value=cur||String(curYr)` — แต่ `""` (ค่า "ทุกปี" ที่ผู้ใช้ตั้งใจเลือก) เป็น falsy ทำให้โดนตีความว่า "ยังไม่ได้ตั้งค่า" แล้วดีดกลับไปปีปัจจุบันทุกครั้งที่ render ใหม่ (แก้ไม่ได้จริง) — แก้โดยเช็ค `hvYearEl.options.length===0` (ยังไม่เคย populate) ถึงจะ default เป็นปีปัจจุบัน ไม่งั้นคงค่าที่ผู้ใช้เลือกไว้เป๊ะๆ รวมถึง "ทุกปี" ด้วย

ทดสอบผ่าน browser: dropdown มีตัวเลือกครบตามปีที่มีข้อมูลจริง, ค่าเริ่มต้นกรองเฉพาะปีปัจจุบัน (1 แถว), เลือก "ทุกปี" แล้ว render ซ้ำหลายรอบยังคงโชว่ครบ (2 แถว) ไม่ดีดกลับ ไม่มี console error

### 16. หน้า "จับกุ้ง" — ตัดตัวเลือก "ปิดบ่อ" ที่ซ้ำซ้อนกับรอบเลี้ยง (แก้แล้ว)
เดิม dropdown "รูปแบบการจับ" มี 2 ตัวเลือก: พาเชียล (จับบางส่วน บ่อยังเลี้ยงต่อ — ข้อมูลเฉพาะตัว ไม่ซ้ำที่ไหน) กับ ปิดบ่อ (กรอกน้ำหนัก/ไซส์/ราคา/วันที่ในฟอร์มนี้ แล้ว `saveHarvest()` จะถามซ้ำให้เปิด `openCropModal` จากการ์ดรอบเลี้ยงเพื่อกรอกชุดเดียวกันอีกรอบ) — ถ้ากด "ยกเลิก" ตอนถาม จะเหลือ record type "ปิดบ่อ" ค้างอยู่ทั้งที่บ่อยังไม่ปิดจริง ข้อมูลไม่ตรงสถานะ

**แก้แล้ว:** ลบ `<option value="ปิดบ่อ">` ออกจาก dropdown เหลือแค่ "พาเชียล" และลบโค้ด confirm/`openCropModal` bridge ใน `saveHarvest()` ออก (dead code เพราะเลือก "ปิดบ่อ" จากฟอร์มไม่ได้แล้ว) — ปิดรอบทำได้ทางเดียวคือปุ่ม "ปิดรอบ" ในการ์ด "รอบเลี้ยง" เท่านั้น

ข้อมูลเก่าที่เคยบันทึกด้วย type "ปิดบ่อ" ไม่ได้ลบทิ้ง — โค้ดแสดงผล/รายงานที่อ่านค่านี้ (สรุปการจับกุ้ง, สถิติจับกุ้ง) ยังคงแสดงได้ปกติ แค่กรอกใหม่เลือกไม่ได้แล้ว

ทดสอบผ่าน browser: dropdown เหลือ 2 ตัวเลือก (ว่าง/พาเชียล), บันทึกพาเชียลสำเร็จปกติ ไม่มี console error

### 15. ต้นทุนค่าไฟรายบ่อ — เปลี่ยนสูตรเป็น 2,500 บาท/วัน×วันเลี้ยง (แก้แล้ว)
**ไฟล์:** `getElecCostForPeriod()` (index.html/TawanFarm_App.html) — จุดใช้: หน้าปิดรอบ (`cm-cost-elec`) + CEO dashboard cost breakdown

เดิมคำนวณจากบิลไฟจริง (`D.electricityBills`) หารเฉลี่ยตามจำนวนบ่อ — เพี้ยนเพราะมีเดือน/วันที่ไม่ได้เลี้ยงกุ้งปนอยู่ในค่าเฉลี่ยรายปีด้วย เปลี่ยนเป็นสูตรคร่าวๆ คงที่: **2,500 บาท/วัน × จำนวนวันที่เลี้ยง** (`ELEC_RATE_PER_DAY=2500`, `dDiff(fromDate,toDate)`) ไม่พึ่งบิลจริงอีกต่อไป

- ลบปุ่ม "💡 ประมาณ" (เดิมอิงค่าเฉลี่ย 19.37 บาท/กก., `estimateElecFromBenchmark()`) ออกจากทั้งโมดัลปิดรอบและแก้ไขรอบที่ปิดแล้ว เพราะช่องหลักดึงอัตโนมัติจากสูตรใหม่อยู่แล้วตลอดเวลา ปุ่มสำรองเดิมเลยฟุ่มเฟือย/อาจทำให้ตัวเลขไม่ตรงกันถ้ากดทับ
- **หน้า "ค่าไฟฟ้า" (กรอกบิลจริงรายเดือน, `D.electricityBills`) ไม่แตะเลยตามที่พี่แจ้ง** — ยังกรอก/ดูประวัติได้ปกติ แค่ไม่ถูกใช้คำนวณต้นทุนต่อบ่อจุดนี้แล้ว เผื่อเอาไว้เทียบ/หาความเชื่อมโยงกันในอนาคต

ทดสอบผ่าน browser: บ่ออายุ 106 วัน → 265,000 บาท ตรงสูตรเป๊ะ, auto-fill ในหน้าปิดรอบถูกต้อง, หน้าค่าไฟฟ้ายังทำงานปกติ ไม่มี console error

### 14. CEO Dashboard — responsive สำหรับ iPad (แก้แล้ว)
เดิม `body.role-ceo .main{max-width:640px}` คงที่ไม่ว่าจอกว้างแค่ไหน ทำให้บน iPad การ์ดบ่อเรียงแถวเดียวยาวๆ เหมือนมือถือ เพิ่ม media query 2 breakpoint (≥700px, ≥1000px) ขยาย `.main`, ปรับ `.ceo-kpi-grid` เป็น 4/5 คอลัมน์, `.ceo-pond-list` เป็น grid 2/3 คอลัมน์, และทำ `.ceo-sheet` เป็นกล่อง 560px กึ่งกลางจอ (ไม่เต็มความกว้างแบบ bottom sheet มือถือ) แทนที่จะขยายเต็มจอเหมือนโทรศัพท์

**บั๊กระหว่างทำ:** วาง media query block ไว้ก่อน base rule (`.ceo-kpi-grid{...}`,`.ceo-sheet{...}`) ในไฟล์ ทำให้ base rule ที่มาทีหลังชนะ cascade แม้ specificity เท่ากัน (ไม่มีผลอะไรเลยตอนทดสอบครั้งแรก) — แก้โดยย้าย media query block ไปอยู่ท้ายสุดของ CEO CSS section แทน ทดสอบยืนยันด้วย `getComputedStyle` ที่ viewport 820px (kpi 4 คอลัมน์, pond list 2 คอลัมน์, sheet กว้าง 560px กึ่งกลาง) และ 390px (กลับเป็นค่าเดิมของมือถือครบ) ไม่มี console error

### 13. CEO Dashboard — แก้ 5 จุดหลังใช้งานจริง (แก้แล้ว)
- **อาหารรวมวันนี้** ไม่ว่างเปล่าแล้ว: `ceoFeedDisplay(p)` ถ้าวันนี้ยังไม่กรอก ดึงบันทึกอาหารล่าสุดก่อนหน้ามาแทน ป้าย "(ล่าสุด)"/"* บางบ่อยังไม่กรอก" บอกให้รู้ว่าไม่ใช่ข้อมูลวันนี้จริง
- **ตัดกราฟเส้น (sparkline) ออกทั้งหมด** — ลบ `ceoSpark()`, canvas ในการ์ดและในแผงรายละเอียด เพราะข้อมูลจุดน้อยกราฟไม่มีประโยชน์จริง
- **ค่าน้ำล่าสุด** เปลี่ยนจาก 3 ค่าตายตัว (pH/DO/Temp เช้า) เป็นวนลูป `PARAMS` โชว์ทุกค่าที่มีการวัดจริงในบันทึกล่าสุด พร้อม ▲/▼/▬ เทียบกับบันทึกก่อนหน้า
- **บั๊กแตะเข้า accordion ไม่ได้ (ต้นทุนสะสม/ค่าน้ำย้อนหลัง/อาหารย้อนหลัง)** — root cause: `.ceo-sheet`/`.ceo-scrim` ใช้ `transform:translateY()` + `opacity` แบบ animated แต่ render ผิดตำแหน่งบนอุปกรณ์จริงจนกดไม่ถึงปุ่ม (ยืนยันด้วยการวัด `getBoundingClientRect` เจอ sheet ค้างอยู่นอกจอ) — **แก้โดยเปลี่ยนกลไกเป็น `display:none/flex` toggle ตรงๆ** ตัด transition/animation ออก แลกความลื่นไหลกับความน่าเชื่อถือ ทดสอบด้วยการยิง `.click()` จริงบน accordion header แล้วเปิด/ปิดได้ถูกต้อง
- **จัดลำดับใหม่**: ข้อมูลบ่อ → ค่าน้ำล่าสุด → การเติบโต (ไม่มีกราฟแล้ว) → accordion 3 หัวข้อ

ผ่านการรีวิว mockup กับพี่ก่อนแก้จริง (ไฟล์ `Mockup และ Prototype/mockup_ceo_dashboard.html` อัปเดตเป็น v5) ทดสอบผ่าน browser: ไม่มี canvas เหลือ, ป้ายอาหารเมื่อวานขึ้นถูกต้อง, ค่าน้ำโชว์ 5 พารามิเตอร์พร้อมลูกศรตรงกับข้อมูลจริง, accordion เปิด/ปิดผ่าน real click event ได้ปกติ, ไม่มี console error
